# dbxignore

Cross-platform Python utility: keeps Dropbox ignore markers (NTFS alternate data streams on Windows; `user.com.dropbox.ignored` xattrs on Linux; `com.dropbox.ignored` xattrs on macOS) in sync with hierarchical `.dropboxignore` files.

## Commands

- `uv sync --all-extras` ŌĆö install
- `uv run pytest` ŌĆö full suite; Windows adds a few ADS-integration tests via `@pytest.mark.windows_only`. If you hit `error: uv trampoline failed to canonicalize script path` or a silent `ModuleNotFoundError`, use `uv run python -m pytest` instead (see Gotchas).
- `uv run pytest -m "not windows_only"` ŌĆö portable subset (what Ubuntu CI runs)
- `uv run pytest -W error::DeprecationWarning` ŌĆö local strict mode (not enforced in CI)
- `uv run ruff check` ŌĆö lint; rule families per `pyproject.toml` `[tool.ruff.lint] select` (don't restate the list here ŌĆö pyproject is the source of truth); line length 100
- `dbxignore <apply|status|list|explain|daemon|install|uninstall>` ŌĆö CLI console script (`cli:main`). `install` / `uninstall` register / remove the daemon with the platform's user-scoped service manager (Task Scheduler on Windows, systemd user unit on Linux, launchd LaunchAgent on macOS). `uninstall --purge` also clears every ignore marker.
- `dbxignored` ŌĆö daemon entry point (`cli:daemon_main`, a standalone `@click.command` with its own `--verbose`/`--version`), launched by the platform's installed service (Scheduled Task / systemd unit / LaunchAgent)
- `python -m dbxignore <subcommand>` ŌĆö equivalent to the console script, via `src/dbxignore/__main__.py`. Useful when the wheel isn't installed (e.g. `uv run python -m dbxignore status`).

## Architecture

`reconcile.reconcile_subtree(root, subdir, cache)` is the single source of truth for rule-driven marker mutations. `cli.apply`, `daemon._dispatch`, and `daemon._sweep_once` all call it ŌĆö never bypass. The lone exception is `cli.uninstall --purge`, which issues an unconditional marker clear (no rule evaluation) while still honoring the `.dropboxignore`-found-marked `WARNING` contract inline. After the marker clear, `--purge` also calls `_purge_local_state()` (removes `state.json`, `daemon.log*`, the per-user state dir if empty) and on Linux additionally calls `linux_systemd.remove_dropin_directory()` to drop any user customization unit drop-ins. Goal: leave no dbxignore-authored artifacts on disk.

Marker I/O is platform-dispatched via `dbxignore.markers`, which at import time re-exports `is_ignored`/`set_ignored`/`clear_ignored` from `_backends/windows_ads.py` (Windows NTFS ADS), `_backends/linux_xattr.py` (Linux `user.com.dropbox.ignored`), or `_backends/macos_xattr.py` (macOS `com.dropbox.ignored` via the `xattr` PyPI package). No other module branches on `sys.platform` for markers. `reconcile._reconcile_path` has symmetric error arms: both **read** and **write** sides catch broad `OSError`, log `WARNING` + append to `Report.errors`, and return a value that lets `reconcile_subtree` continue. Read returns `None` (vanished/unreadable path ŌĆö don't drive subtree pruning); write returns `currently_ignored` (matching `PermissionError`'s arm, item #41 ŌĆö last-known marker state drives subtree pruning when an already-marked directory's write fails; returning `None` would re-walk into a subtree the filesystem can't mark and spam `WARNING`s per child). The write arm preserves a specific log message for `ENOTSUP/EOPNOTSUPP` ("Filesystem does not support ignore markers...") because that's a sysadmin-actionable distinction; other errnos log a generic `errno=NN` line. Original design (pre-#21) was narrow-by-design on both sides; #21 widened the read arm to handle transient EIO on network drives without killing the per-root sweep, and PR #128 widened the write arm symmetrically for the same reason. Real bugs (non-`OSError` exceptions) still propagate.

`daemon._sweep_once` fans `reconcile_subtree` out across roots via `ThreadPoolExecutor` (one worker per root). Safe because reconcile reads the cache lock-free (each `.get()` is GIL-atomic) and writes per-file ignore markers on disjoint paths. Multi-step traversals like `_applicable` are *not* transactional ŌĆö between `.get()` calls a debouncer-thread mutation can change which ancestor's rules apply, so a single sweep tick may match against a slightly-stale ancestor view. Downstream is convergent: the next watchdog event or hourly sweep recovers, so the design accepts the drift instead of paying the per-`_applicable` lock cost. `RuleCache._rules` is guarded by a `threading.RLock` ŌĆö any mutation (`load_root`, `reload_file`, `remove_file`, or the stale-purge iteration in `load_root`) must go through it, otherwise the debouncer thread can race with the main-thread sweep. If you add cross-root shared state to `RuleCache` or reconcile, revisit this.

`rules.RuleCache` stores one `_LoadedRules(lines, entries, mtime_ns, size)` per `.dropboxignore`. `entries` is a list of `(source_line_index, pathspec.Pattern)` pairs and is the single source of truth for both `match()` and `explain()`.

The static conflict detector lives in `rules_conflicts.py` (extracted from `rules.py`); `RuleCache._recompute_conflicts` is its sole production caller. Public symbols (`Conflict`, `_detect_conflicts`) are re-exported from `rules.py` for back-compat.

`rules._load_if_changed` skips reparse when a `.dropboxignore`'s `mtime_ns` and `size` both match the cached values ŌĆö that's why `_LoadedRules` carries stat fields. The sweep path (`load_root`) uses it; watchdog-driven `reload_file` bypasses it because an explicit event is authoritative.

The daemon's watchdog events are classified (`_classify` ŌåÆ `EventKind.{RULES,DIR_CREATE,OTHER}`) and funneled through `Debouncer` before `_dispatch` runs `reconcile_subtree`. `DEFAULT_TIMEOUTS_MS` per kind is overridable via `DBXIGNORE_DEBOUNCE_{RULES,DIRS,OTHER}_MS`. One deviation from the funnel: `_WatchdogHandler.on_any_event` fast-paths a matched `DIR_CREATE` (the directory's path already matches a cached rule) by calling `reconcile_subtree` synchronously and skipping `Debouncer.submit` ŌĆö this narrows the race window where Dropbox sees the new directory before its marker lands. Other event kinds (RULES, OTHER) and unmatched DIR_CREATEs still go through the debouncer. Trade-off: a queued-but-unprocessed RULES event that would invalidate the match means the bypass marks a path the next `reconcile_subtree` (driven by that RULES event) clears ŌĆö bounded transient false-positive.

`install/` is a package with `__init__.py` exposing `install_service()` / `uninstall_service()` dispatchers, plus `windows_task.py` (schtasks XML generation + invocation), `linux_systemd.py` (writes `~/.config/systemd/user/dbxignore.service`, runs `systemctl --user daemon-reload && enable --now`), and `macos_launchd.py` (writes `~/Library/LaunchAgents/com.kiloscheffer.dbxignore.plist`, bootstraps via `launchctl bootstrap gui/<uid>`). `cli.install` / `cli.uninstall` delegate only to the package dispatcher; don't call backend modules directly from the CLI layer.

`state.is_daemon_alive(pid: int | None) -> bool` is the canonical liveness check ŌĆö verifies BOTH that the PID exists AND that the process at that PID is plausibly a dbxignore daemon (matches `"python"` or `"dbxignored"` in the process name; without the name guard, a recycled PID claimed by an unrelated process registers as alive). `cli.status` and `daemon._is_other_live_daemon` both delegate to it; new code touching daemon liveness should too.

Destructive CLI verbs that race the daemon's reconcile loop (today: `cli.clear`) refuse-to-run when `state.is_daemon_alive(s.daemon_pid)` is True ŌĆö the daemon's rule-reload reconcile fires within seconds of a `.dropboxignore` save, and the hourly recovery sweep reapplies markers wholesale, so a client-side mutation gets silently undone within an hour at worst. Pattern: `--force` overrides the daemon-alive guard for known short-window tests, `--yes` skips the confirmation prompt for scripted use, `--dry-run` previews. Both directions are destructive and both warrant a confirmation guard: `clear` triggers Dropbox to upload previously-ignored paths (potentially gigabytes for a `node_modules`-class subtree); `apply` (and any rule edit that flows through the daemon) causes Dropbox to delete previously-synced paths from the cloud and propagate the deletion to every linked device. The mental model that "ignored = stop syncing on this device" is wrong ŌĆö Dropbox treats the marker as authoritative and removes the cloud copy, leaving the local copy as the only surviving copy until the marker is cleared. Today only `clear` carries the `--yes`/confirmation ceremony; the daemon-alive guard does not generalize to `apply` (the daemon performs `apply`'s operation continuously, so racing it is normal usage), but the confirmation-on-first-mark prompt and `--yes` opt-out belong on `apply` for the same reason they belong on `clear`.

CLI fatal-error exits use `sys.exit(2)` project-wide (~26 callsites in `cli.py`). The verdict-driven commands `explain` / `check-ignore` (PR #127) use `0` (ignored) / `1` (not ignored) for the verdict surface AND `2` for fatal (no Dropbox roots, etc.) ŌĆö preserving the project convention rather than git's `128`. New CLI commands should follow the `2`-for-fatal pattern; full git-parity (`128`) would be a separate exit-code modernization PR.

## Gotchas

- pathspec 1.0.4: subclass `GitIgnoreSpecPattern`, not deprecated `GitWildMatchPattern`.
- pathspec 1.0.4: `spec.check_file(path)` returns `CheckResult(include, index, file)` ŌĆö use when you need pattern-level verdicts beyond a bare bool.
- pathspec: `pattern.match_file()` is public; `pattern.regex.match` is private API.
- pathspec: directory-only rules (`node_modules/`) require trailing `/` on the tested path string.
- pathspec follows gitignore's column-0 comment rule: a line with leading whitespace before `#` (e.g. `"   #literal"`) is an *active pattern*, not a comment. `rules._build_entries`'s filter checks `raw.startswith("#")` (not `raw.strip().startswith("#")`) so the line is correctly classified as active. The count-mismatch fallback at the bottom of `_build_entries` is now defensive scaffolding for future pathspec-version drift; under the gitignore-correct filter, fast-path counts match in practice.
- pathspec is liberal: gitwildmatch accepts `[invalid`, `**/*.{a,b`, etc. without raising. To test code that catches `(ValueError, TypeError, re.error)` from `_build_spec`, monkeypatch `rules._build_spec` to raise rather than searching for an input that triggers it.
- `cli.py` deliberately uses `from dbxignore import rules` (module attribute access) so `cli.generate` and `cli._apply_from_gitignore` call `rules._build_spec(...)` via lookup at use time. Tests rely on this: `monkeypatch.setattr(rules, "_build_spec", ...)` reaches the call site. Switching to `from dbxignore.rules import _build_spec` (symbol import) would silently break `tests/test_cli_generate.py::test_generate_invalid_pattern_writes_nothing`.
- click + rich_click: the project sets `rich_click.TEXT_MARKUP = "markdown"`, which strips unescaped backticks from command docstrings before they reach `--help` output. To preserve literal backticks (e.g. for help-text assertions like `"Alias of \`explain\`"` in `tests/test_cli_status_list_explain.py::test_check_ignore_help_distinguishes_from_explain`), escape them in the docstring source. Surfaced in PR #127's `check-ignore` alias.
- `_backends/windows_ads` uses `open(r"\\?\path:com.dropbox.ignored")` directly ŌĆö `\\?\` prefix mandatory for >260-char paths.
- NTFS is case-insensitive; `_CaseInsensitiveGitIgnorePattern` prepends `(?i)` to compiled regexes.
- `.dropboxignore` files are never marked ignored ŌĆö guarded in `match()` and `explain()`; `reconcile._reconcile_path` clears any ADS marker it finds on one and logs at `WARNING` (spec contract ŌĆö don't silence it in a refactor).
- `rules.match/explain` and `markers.{is,set,clear}_ignored` all require **absolute** paths and raise `ValueError` on relative ones. Resolve at the CLI/daemon boundary, never inside the cache or markers layer ŌĆö `Path.resolve()` on Windows is a per-call syscall that dominated sweep wall-clock before.
- `daemon._resolve_under_roots(raw_path, roots)` is the canonical helper for the watchdog path: returns `(root, resolved_path)` if the raw path is under a watched root, else `None`. The order matters ŌĆö `find_containing` runs *before* `Path.resolve()`, so events outside any root pay zero resolve cost. Used by `_classify` (`event.src_path`) and `_dispatch` (`event.dest_path` on `moved` events). Don't invert the order in a future refactor: that's exactly the cost item #43 hoisted out.
- `watchdog.observers.Observer` is a platform-conditional factory function, not a class ŌĆö annotating a parameter as `Observer` fails mypy with `[valid-type]`. For type annotations, import `BaseObserver` from `watchdog.observers.api` and (since `from __future__ import annotations` is in effect, ruff TC002) place it under `if TYPE_CHECKING:`. The runtime construction `observer = Observer()` stays as today. Surfaced when adding `daemon._start_observer_or_exit(observer: BaseObserver)` in PR #125.
- `_rel_path_str(ancestor, path, is_dir)` takes `is_dir` as a parameter, not because the function needs it polymorphically but because `match()` / `explain()` compute `path.is_dir()` once per call and pass the bool down to every ancestor iteration ŌĆö the directory-only-pattern check (`node_modules/`) was repeating the syscall D times for the same path when D ancestor `.dropboxignore` files applied. Same shape as the `Path.resolve()` rule above: stat syscalls in the per-file sweep loop are expensive in aggregate, hoist them to the per-call layer.
- `daemon._configured_logging()` is a context manager: it snapshots the `dbxignore` logger on enter and restores handlers/propagate/level on exit. On Linux it installs two handlers ŌĆö `RotatingFileHandler(daemon.log)` plus `StreamHandler(sys.stderr)` so systemd-journald captures the same records ŌĆö on Windows only the file handler.
- Test implications of the above: tests that count handlers must branch on `sys.platform`. `run()` wraps its body in `_configured_logging`, so tests calling `daemon.run()` don't need to hand-restore logger state ŌĆö but if you mock the context manager out, use `contextlib.nullcontext` (see `test_daemon_singleton.py`).
- Use `datetime.UTC`, not `timezone.utc` (ruff UP017).
- Test helpers (`FakeMarkers`, `fake_markers` fixture, `write_file` fixture) live in `tests/conftest.py` and are auto-available to every test module.
- Log-contract tests use `caplog.at_level(logging.WARNING, logger="dbxignore.<module>")` ŌĆö narrow to the submodule that emits the log (see `tests/test_reconcile_edges.py`).
- Windows-only tests: set `pytestmark = pytest.mark.windows_only` at module level and guard with `if sys.platform != "win32": pytest.skip(..., allow_module_level=True)` so non-Windows collection skips cleanly. Same shape for `linux_only` and `macos_only`.
- Daemon/sweep tests that trigger state writes: `monkeypatch.setattr(state, "default_path", lambda: tmp_path / "state.json")` redirects the persisted state off the real per-user state dir and keeps the test hermetic.
- Per-user state dir varies: `%LOCALAPPDATA%\dbxignore\` on Windows, `$XDG_STATE_HOME/dbxignore/` (fallback `~/.local/state/dbxignore/`) on Linux, `~/Library/Application Support/dbxignore/` on macOS. Single source of truth is `state.user_state_dir()`. macOS additionally splits the daemon log into `~/Library/Logs/dbxignore/` per Apple's app-data conventions ŌĆö `state.user_log_dir()` exposes this; on Windows + Linux it falls through to `user_state_dir()`. `daemon._log_dir()` reads from `user_log_dir()`. `cli._purge_local_state()` cleans both dirs on darwin. The v0.2-era Linux legacy-path fallback was removed in v0.3.
- `state.write()` writes to `state.json.tmp` then `os.replace`s into place (POSIX-atomic on Linux; `MoveFileExW(MOVEFILE_REPLACE_EXISTING)` on Windows). Direct `path.write_text()` would truncate-then-write ŌĆö a SIGKILL or power loss between the two leaves a zero-length / partial `state.json`; `_read_at` then returns `None` (`JSONDecodeError`) and `daemon.run`'s singleton check thinks no prior daemon exists, allowing two concurrent daemons. Apply the same temp+replace pattern for any future state-file writes; `_purge_local_state` cleans a leaked `state.json.tmp` if one is found.
- Root-discovery test seams, by layer: CLI tests monkeypatch `cli._discover_roots`; daemon tests monkeypatch `daemon.roots_module.discover`; `roots.discover()` unit tests use `_stage_info(monkeypatch, tmp_path, fixture_name)` which sets `APPDATA` (Windows) or `HOME` (Linux) to point at a staged `Dropbox/info.json` / `.dropbox/info.json`. Pick the layer that matches the code path under test.
- Linux xattrs vanish silently through common operations: `cp` without `-a`, cross-filesystem `mv`, most archivers, and `vim`'s default save-via-rename. The watchdog event stream + hourly sweep are the recovery mechanism ŌĆö don't add a preservation wrapper, the design intentionally leans on reconcile.
- xattr-based backends avoid following symlinks (mirrors `os.walk(followlinks=False)` in reconcile). Per platform:
    - **Linux**: kernel refuses `user.*` xattrs on symlinks (EPERM). `set/clear_ignored` raise `PermissionError` ŌåÆ reconcile logs WARNING and skips. `is_ignored` returns False (ENODATA).
    - **macOS**: kernel allows `com.dropbox.ignored` on symlinks via NOFOLLOW; the link itself gets marked. `reconcile._reconcile_path`'s `PermissionError` arm is dormant.
    - **Windows**: ADS attaches to the symlink reparse point.
   Tests asserting symlink behavior must branch on `sys.platform`.
- `roots.discover()` branches on `sys.platform`: Windows checks `%APPDATA%\Dropbox\info.json` (per-user installer) AND `%LOCALAPPDATA%\Dropbox\info.json` (per-machine "install for all users") in priority order ŌĆö the per-machine path was added when a v0.4 alpha tester reported "No Dropbox roots found" on a per-machine install; Linux and macOS use `~/.dropbox/info.json` (folded into one arm because Dropbox's desktop client uses the same path on both). Same JSON schema across all platforms. The `_info_json_paths()` helper returns a `list[Path]` of candidates (empty on unsupported platforms or when env vars are missing); `discover()` iterates and uses the first existing file, or warns and returns `[]` if none exist.
- `roots.discover()` honors `DBXIGNORE_ROOT` as a pre-`info.json` escape hatch for non-stock Dropbox installs. Set to an existing absolute path ŌåÆ `[Path(env)]`, bypassing `info.json` (and the platform check ŌĆö useful on platforms where `_info_json_path()` returns `None`). Set to a nonexistent path ŌåÆ WARNING + `[]`. Empty string ŌåÆ treated as unset. Single-root only; multi-account setups with an override have to pick one.
- `RuleCache` runs a static conflict detector at every mutation (`load_root`, `reload_file`, `remove_file`). Negations whose literal path prefix lives under a directory matched by an earlier include rule are recorded in `_conflicts`, and their `(source, line_idx)` tuple is added to `_dropped`.
- Downstream consumers diverge: `match()` and reconcile ignore anything in `_dropped`; `explain()` still returns dropped matches but sets `Match.is_dropped=True` so CLI/log formatters can annotate them. Semantic reason: Dropbox's ignored-folder inheritance makes negations inert under ignored ancestors, so we don't pretend they're in effect.
- Local pytest gotcha: `uv run pytest` may not pick up the editable install after `uv pip install -e .`. Two failure modes seen: silent `ModuleNotFoundError: No module named 'dbxignore'`, and `error: uv trampoline failed to canonicalize script path`. Same workaround in both cases ŌĆö use `uv run python -m pytest`, which invokes the interpreter that has the `.pth` file. CI doesn't hit this (fresh provisioning).
- Windows install gotcha: `uv tool install git+...` can fail with `ERROR_CLOUD_FILE_INCOMPATIBLE_HARDLINKS` (os error 396) when `%AppData%` is OneDrive-synced via Files On-Demand. uv hardlinks files from `%LOCALAPPDATA%\uv\cache\` into `%APPDATA%\uv\tools\<pkg>\Lib\site-packages\` by default; the Cloud Files API refuses hardlinks on placeholder files. Workaround documented in the README's Windows install section: `uv tool install --link-mode=copy git+...` (or `$env:UV_LINK_MODE = "copy"`). CI doesn't hit this ŌĆö the GitHub Windows runner has no cloud-synced AppData.
- Check a GitHub Action's runtime before bumping: `gh api "repos/<owner>/<repo>/contents/action.yml?ref=<tag>" --jq .content | base64 -d | grep using`. `using: 'nodeNN'` = Node-based (subject to runtime deprecation); `using: composite` = shell-script orchestrator (immune).
- The `xattr` PyPI package's module-level `getxattr`/`setxattr`/`removexattr` use `symlink=True` to operate on the link itself (NOT `options=xattr.XATTR_NOFOLLOW` as the underlying macOS C API would suggest). The kwarg is named `symlink` and its boolean is "operate on the link, don't follow." Verified the hard way during PR 55's first macOS CI run ŌĆö the v0.4 spec had projected from Apple's libc docs and was wrong. Anyone touching `_backends/macos_xattr.py` should match the package's actual signature, not the platform's.
- macOS daemon installer (`install/macos_launchd.py`) uses modern `launchctl bootstrap gui/<uid> <plist>` and `bootout gui/<uid>/<label>` ŌĆö not legacy `load -w` / `unload -w`. Modern bootstrap requires the user to have logged into the GUI at least once since the last reboot (the GUI domain isn't initialized until login). SSH-on-fresh-boot installs fail with `Bootstrap failed: 5: Input/output error`. Documented in the README; no fallback to legacy syntax (workaround is "log into GUI, retry").
- `install/_common.detect_invocation()` is shared between `linux_systemd.py` and `macos_launchd.py`. Both lazy-import it inside their `install_*()` functions. When monkeypatching for tests, patch at the **import site** (`dbxignore.install.linux_systemd.detect_invocation` or `dbxignore.install.macos_launchd.detect_invocation`), not at the source (`_common.detect_invocation`) ŌĆö `from ... import` creates a local binding in the consuming module, and that binding is what needs intercepting.
- **macOS Dropbox sync modes ŌĆö the two attribute names.** `_backends/macos_xattr.py` writes one of two xattrs depending on which sync stack is active. Legacy mode (Dropbox folder at `~/Dropbox`, synced by Dropbox's own daemon) recognizes `com.dropbox.ignored`. File Provider mode (Dropbox folder at `~/Library/CloudStorage/Dropbox/`, synced by Apple's File Provider extension via `DropboxFileProvider.appex`; default for installs since 2023) recognizes `com.apple.fileprovider.ignore#P` ŌĆö the `#P` suffix is Apple's "persistent across reboots" marker convention for File Provider attributes.
- **macOS sync-mode detection algorithm.** `_detect()` is **path-primary, pluginkit-disambiguating** and returns `(attr_names, summary)`. `_read_dropbox_paths_from_info()` reads `~/.dropbox/info.json`'s `path` field (multi-account aware ŌĆö info.json can list `personal` + `business`); `_pluginkit_extension_state()` queries `pluginkit -m -A -i com.getdropbox.dropbox.fileprovider` for the framework-level state. Truth table: extension explicitly disabled ŌåÆ `[ATTR_LEGACY]` regardless of path; any account path under `~/Library/CloudStorage/` ŌåÆ `[ATTR_FILEPROVIDER]`; path elsewhere + extension allowed ŌåÆ `[ATTR_FILEPROVIDER]` (external-drive eligibility-gated case); pluginkit unknown + no decisive path signal ŌåÆ `[ATTR_LEGACY, ATTR_FILEPROVIDER]` (write-both defensive case, followup item 58); otherwise ŌåÆ `[ATTR_LEGACY]`. The decision is cached on `_decision_cache` (module-local `tuple[list[str], str] | None`) after first call so the per-file reconcile loop doesn't re-invoke pluginkit + re-parse info.json. The companion `detection_summary()` function returns the human-readable `<mode>: <reason>` string that the daemon logs at INFO startup and `dbxignore status` echoes on darwin (followup item 37). The legacy `_detected_attr_name() -> str` is retained as a back-compat shim returning `_detected_attr_names()[0]` for older tests. Tests mock `subprocess.run` via the `_fake_pluginkit` helper and stage info.json via `_stage_dropbox_info` in `tests/test_macos_xattr_unit.py`.
- **Dual-attribute mode behavior** (followup item 58). When `_detected_attr_names()` returns both names, `is_ignored` iterates and short-circuits True on the first non-empty hit (so legacy users don't pay two getxattr calls per file); `set_ignored` writes both names sequentially and a partial-failure on the second propagates with the second's exception, mirroring the single-attr "either fully successful or raises" contract; `clear_ignored` removes both with per-attribute ENOATTR no-op so a half-marked path that only ever had one name written gets cleaned up regardless. The trade is a stray attribute on the inactive sync stack (metadata cleanliness) versus a silent no-op on the active stack (correctness).
- **`markers.detection_summary()` cross-platform facade** (followup item 37). Returns a `<mode>: <reason>` string on darwin and `None` on Windows/Linux ŌĆö the latter two are single-attribute platforms with nothing to detect. Callers (`daemon.run` for the INFO log line, `cli.status` for the `sync mode:` echo) gate on `if summary is not None:` so the same code path works on every platform.
- **Sync-mode detection is path-primary, pluginkit-disambiguating.** PluginKit registration is a system-level fact ("does macOS know about the appex?"); the user-level fact ("which mode is *this account* in?") lives in info.json. v0.4.0a4 conflated the two and misdetected users who had Dropbox.app installed but had declined the File Provider migration; a5 split the signals.
- **`pluginkit -m -A -i <bundle-id>` output format**: one line per matching extension with a status-prefix character. Whitespace = default-state (registered, untoggled); `+` = user explicitly enabled; `-` = user explicitly disabled. `_pluginkit_extension_state()` returns `"disabled"` for `-`, `"allowed"` for whitespace/`+`, `"not_registered"` when no match, `"unknown"` on subprocess error. Detection treats `"disabled"` as a hard override (legacy regardless of info.json path) and `"allowed"` as permissive, combined with path data.
- Subprocess-using code (e.g. `_pluginkit_extension_state` on macOS) tests via `monkeypatch.setattr("subprocess.run", fake_fn)` rather than mocking the calling function. Template: `_fake_pluginkit` helper in `tests/test_macos_xattr_unit.py` ŌĆö accepts `stdout=` for the success path or `side_effect=` for `FileNotFoundError`/`TimeoutExpired`. Future install-backend tests (schtasks, systemctl, launchctl) should use the same shape.
- Packaged non-.py data files (currently `src/dbxignore/templates/default.dropboxignore`, used by `cli.init`) ship automatically because hatchling's `packages = ["src/dbxignore"]` includes everything under the package directory, including subdirs and non-.py files. Access at runtime via `importlib.resources.files("dbxignore.<subpkg>").joinpath("<file>").read_text(encoding="utf-8")` ŌĆö requires an `__init__.py` in the subpkg directory. Verify packaging on a fresh add by `uv build && python -c "import zipfile; z = zipfile.ZipFile(<wheel>); print([n for n in z.namelist() if '<subpkg>' in n])"` before relying on it.
- Tests asserting `dbxignore status` or `cli.clear` daemon-alive behavior should `monkeypatch.setattr(state, "is_daemon_alive", lambda pid: <bool>)` rather than relying on `daemon_pid=os.getpid()` matching the daemon-name guard. The pytest process name varies by host: Linux + `uv run pytest` shows up as `pytest`, Windows + `uv run python -m pytest` as `python.exe`, and `is_daemon_alive` only matches `"python"` or `"dbxignored"` substrings ŌĆö Linux runs land in `state=not_running` if the test relies on `os.getpid()`. Same shape as the `legacy_mode` fixture lesson for the macOS xattr tests (PR #97's first Ubuntu CI failure).
- PowerShell `Set-Content -Encoding utf8` writes UTF-8-with-BOM on Windows PowerShell 5.1 but BOM-less on PS 7+. Any new PS script under `scripts/` should `#requires -Version 7.0` at the top ŌĆö gates against the divergence with a clear error message rather than silently producing different file bytes (would break the byte-for-byte 4h `generate` assertion in `manual-test-windows.ps1` if the runner happened to launch under PS 5.1).
- `mypy.ini` in the repo root (or `~/.mypy.ini`) silently overrides `pyproject.toml`'s `[tool.mypy]` block ŌĆö mypy's config-file precedence is `mypy.ini` > `.mypy.ini` > `pyproject.toml`. Project config lives in `pyproject.toml`; `mypy.ini` is `.gitignored` for local scratch. If a local `mypy.ini` ever appears with a `[mypy]` section, the strict block in pyproject.toml stops applying and mypy reports a small subset of the real errors.
- Cross-platform `# type: ignore` codes differ by what's installed: `psutil` and `xattr` produce `[import-not-found]` on Windows (not installed) but `[import-untyped]` on Linux/macOS (installed, no stubs). Same shape for POSIX-only `os.getxattr`/`setxattr`/`removexattr`/`getuid` (`[attr-defined]` on Windows; the attr exists on Linux/macOS). Use the combined form `# type: ignore[import-not-found, import-untyped, unused-ignore]` (or `[attr-defined, unused-ignore]`) so the ignore covers whichever code fires AND silences `warn_unused_ignores` on the platforms where it doesn't.
- `uv run mypy .` repo-wide reports pre-existing `[attr-defined]` errors in `tests/conftest.py` (`Module "dbxignore" has no attribute "cli"`, `"reconcile"`, etc.). Cause: strict mypy + `no_implicit_reexport=true` rejects the `from dbxignore import <submodule>` pattern because `__init__.py` doesn't explicitly re-export the submodules. Adding `py.typed` doesn't help (the package is already analyzed via the editable install). PRs work around with `--no-verify` after scoped checks (`mypy <PR-touched-files>`) pass. Fix candidate (deferred): add `from . import cli, daemon, install, markers, reconcile, roots, rules, state` to `src/dbxignore/__init__.py` ŌĆö would need to verify it doesn't break import-time eagerness assumptions.
- Two independent `dev` groups live in `pyproject.toml`: `[project.optional-dependencies].dev` (PEP 621 extras, ships in the wheel for `pip install dbxignore[dev]`) and `[dependency-groups].dev` (PEP 735, local-only). `uv sync --all-extras` hydrates both; plain `uv sync` only hydrates the second. CI uses `--all-extras`. Keep critical test-runtime deps (`pytest-timeout`, the conditional `xattr`) mirrored in both groups so `uv sync` plain doesn't ship a broken environment.
- `pytest.skip(...)` is typed as `NoReturn`, but mypy strict under host=darwin doesn't always pick up the flow narrowing when skip is the tail of an `if/elif/else` chain ŌĆö variables bound in the if/elif arms are flagged "not defined" at use sites that follow the chain. Workaround: skip up front (`if sys.platform not in supported: pytest.skip(...)`), then use a 2-arm `if/else` for the supported platforms.
- Windows + Dropbox-synced working tree: `uv sync` / `uv pip install` editable-install steps hit `os error 32` ("file is being used by another process") on `.venv/Lib/site-packages/dbxignore-*.dist-info` while Dropbox is mid-sync. Workaround: `UV_LINK_MODE=copy` (env var or `--link-mode=copy`). The pre-commit hook's `uv run` invocation inherits the same problem; if the hook fails on the editable-install step (not on the actual lint/format/mypy check, which all show "Passed" in the hook output) it's the file lock, and `--no-verify` is acceptable for that specific commit when the manual `uv run` checks all pass cleanly. Same family as the existing `uv tool install` Cloud Files gotcha but for the local `.venv`.
- `--no-verify` skips ALL pre-commit hooks at once, including commit-check at the `commit-msg` stage ŌĆö not just the lint/format/mypy hook that's failing on the file-lock above. When using `--no-verify`, run commit-check manually against each new commit subject before pushing: `for sha in $(git log origin/main..HEAD --format='%h'); do git log -1 --format='%B' $sha | commit-check -m /dev/stdin; done` ŌĆö otherwise an over-72-char subject (or any other `cchk.toml` violation) lands locally and CI's commit-check action rejects the PR. CI runs commit-check across the full `origin/main..HEAD` range, so the workaround needs to validate every commit, not just `HEAD`.

## Git workflow

- Never commit directly to `main`. Work on a topic branch and open a PR ŌĆö that's what triggers `.github/workflows/test.yml` (the platform-gated test tiers `pytest -m windows_only`, `pytest -m linux_only`, and `pytest -m macos_only` **only run in CI**) and `.github/workflows/commit-check.yml` (commit-message + branch-name validation). A local `uv run pytest` can only exercise one platform, so a single green local run is not a merge gate. The PR matrix is.
- **`cchk.toml` at repo root is the single source of truth** for allowed commit types (Conventional Commits, see `allow_commit_types`), branch types (Conventional Branch, see `allow_branch_types`), and the subject-length cap. Don't restate those lists here or elsewhere ŌĆö reference the file so it can't drift. Local enforcement is optional but encouraged: `uv tool install pre-commit && pre-commit install --hook-type commit-msg --hook-type pre-push` wires the same rules at commit/push time. CI re-runs them on every PR via `commit-check/commit-check-action@v2.6.0`.
- **Pre-flight commit-check against every commit, not just HEAD.** CI runs `commit-check` over the full `origin/main..HEAD` range ŌĆö a check that only validates the planned PR title can pass locally while an intermediate commit fails CI (PR #12: a commit description starting with `--` tripped commit-check's regex). Loop over each subject before pushing.
- Branch names follow `<type>/<slug>` where `<type>` is from `cchk.toml`'s `allow_branch_types` and `<slug>` is lowercase-alphanumeric + hyphens. Note two asymmetries with commit subjects: (1) the branch prefix `feature/` is the long form while the Conventional Commits subject tag `feat:` is the short form; (2) `allow_branch_types` is a **strict subset** of `allow_commit_types` ŌĆö `docs/`, `style/`, `refactor/`, `perf/`, `test/`, `build/`, `ci/`, `revert/` are valid commit types but NOT valid branch prefixes. Use `chore/` for those categories of work. Examples: `feature/v0.2-linux`, `fix/v0.2-followups-2-5`, `fix/v0.2-followup-1-linux-xdg-paths`.
- Commit subjects follow Conventional Commits: `<type>(<scope>): <description>` where `<type>` is from `cchk.toml`'s `allow_commit_types`. Scope tags mirror package names or doc categories, not ticket numbers. `!` before the colon ŌĆö or a `BREAKING CHANGE:` footer ŌĆö signals a breaking change.
- Split commits along revertability lines: a code change and a doc-only backlog update belong in separate commits because they could plausibly be reverted at different times. PR #4 is the template ŌĆö one `feat` commit for the behavior change, one `docs` commit for the new follow-up entries.

## Release

- `.github/workflows/test.yml` runs ruff + the portable pytest subset on `ubuntu-latest`, `windows-latest`, and `macos-latest` for every push/PR. Each platform leg additionally runs its own `pytest -m <platform>_only` step.
- Push tag `v*` ŌåÆ `.github/workflows/release.yml` builds wheel + `dbxignore.exe` / `dbxignored.exe` (Windows, via `pyinstaller/dbxignore.spec`) + `dbxignore` / `dbxignored` arm64 Mach-O (macOS, via `pyinstaller/dbxignore-macos.spec`) and publishes to two destinations: GitHub Release (auto, tag-gated; pre-release flag sourced from the `classify-tag` job) and PyPI (only wheel/sdist ship to PyPI ŌĆö Mach-O and `.exe` binaries are GitHub-Release-only). `hatch-vcs` derives the version from the tag ŌĆö no manual `pyproject.toml` bump needed. PyPI uses Trusted Publishing; no API token secret stored.
- A `classify-tag` job is the single source of truth for "is this a pre-release tag?" ŌĆö it parses the tag in shell against PEP 440 patterns (`aN`, `bN`, `rcN`, `.devN`) and emits an `is_prerelease` boolean output. `publish-github`'s `prerelease:` flag and `publish-pypi`'s `if:` clause both read from that output, so they cannot disagree. **Do NOT use `contains(github.ref, '-a')`-style checks** ŌĆö PEP 440 has no hyphen separator (alphas are `0.4.0a1`, not `0.4.0-a1`), so substring checks against hyphenated SemVer-style markers always evaluate False against the tags `hatch-vcs` produces. v0.4.0a1 misfired through PyPI for exactly this reason; PR #62 introduced the `classify-tag` job to fix it.
- PyPI publish has two safety layers: (1) PEP 440 pre-release tags skip the `publish-pypi` job entirely via the `classify-tag`-driven `if:` clause ŌĆö they're never queued, so there's nothing to accidentally approve; (2) release tags enter the job and pause at the `pypi` GitHub environment gate, which requires a maintainer approval click in the GitHub UI before the OIDC upload fires. The two layers cover different threat models: the `if:` exclusion prevents pre-releases from hitting PyPI even if someone clicks approve; the environment gate prevents release tags from auto-publishing without human review. To pre-release without PyPI: tag with PEP 440 pre-release suffix (`v<X.Y.Z>a<N>`, `v<X.Y.Z>rc<N>`, `v<X.Y.Z>.dev<N>`). To release: tag `v<X.Y.Z>` and approve at the gate. Note: `.postN` (post-release) is intentionally NOT a pre-release ŌĆö post-release tags publish to PyPI.
- Before tagging, sanity-check wheel metadata: `uv build && unzip -p dist/*.whl '*.dist-info/METADATA' | head` ŌĆö confirms `Name: dbxignore` and the expected `Version:` are in the wheel before a misnamed upload hits PyPI (immutable; yank-only recovery).
- `CHANGELOG.md` at repo root follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/): new entries accrue under an `[Unreleased]` heading (add it at the top when the first post-release change lands) and roll into a version heading with its release date when the tag goes out. Hand-crafted per-version release bodies live under `docs/release-notes/v<X.Y.Z>.md` for use with `gh release edit v<X.Y.Z> --notes-file docs/release-notes/v<X.Y.Z>.md` after the workflow publishes.
- This project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html). Pre-1.0, breaking changes ride MINOR bumps with explicit **Breaking** callouts in the CHANGELOG ŌĆö v0.2.0 introduced two (broadened `--purge`, changed `explain` format). Post-1.0, breaking changes will bump MAJOR.
- Watch a long-running release.yml run via `gh run watch <run-id> --exit-status` invoked with `run_in_background: true`. The agent gets a notification when the run hits a terminal state (success/failure/cancelled). Don't poll via Bash sleep loops ŌĆö the harness blocks long leading sleeps for exactly this reason.

## Manual test scripts

`scripts/manual-test-{ubuntu-vps,macos,windows}.{sh,ps1}` are end-to-end smoke tests requiring a live Dropbox install. CI does not run them ŌĆö they're for release-prep verification of behavior the unit tests can't reach (real markers on a real synced tree, real service-manager registration, real cross-process daemon interaction). All three follow the same phase numbering: 0 (preflight), 1 (verify Dropbox), 2 (install dbxignore), 3 (CLI surface), 4aŌĆō4f (reconcile), 4.5 / 4gŌĆō4n (extended CLI), 5 (daemon), 6 (uninstall), 7 (cleanup).

- **When a PR adds or changes user-visible CLI surface** ŌĆö new subcommand, new flag with marker/Dropbox side effects (`--yes`, `--dry-run`, `--purge`, `--force`), new stderr warning, or behavior change unit tests can't fully cover ŌĆö extend all three scripts under Phase 4.5 (`phase_extended_cli` in bash, `Test-ExtendedCli` in PowerShell). Each case carries an inline `# 4X ŌĆö <description> (PR #NNN)` provenance comment so future contributors can map test cases back to their PRs. Phase 4.5 was backfilled in PR #114 (~190 LOC per bash script + a 734 LOC new Windows script) after coverage gaps from PRs #100, #102, #103, #107, #108 surfaced ŌĆö catching at PR-time is ~30 LOC per script per case; backlog catch-up is significantly more expensive. Backlog item #75 tracks extracting the duplicated Phase 4.5 logic into a cross-script helper; until that lands, keep the three scripts in sync by hand.
- **Cross-platform behavioral divergences** branch per-script with inline comments. Canonical examples: 4e symlinks (Linux refuses `user.*` xattrs with EPERM ŌåÆ WARNING; macOS allows via the NOFOLLOW path; Windows attaches ADS to the reparse point ŌåÆ marks silently). Don't paper over these ŌĆö the manual scripts are where the divergences are surfaced for testers, and each platform script inverts the assertion accordingly.
- **PowerShell scripts** (`scripts/manual-test-windows.ps1`) start with `#requires -Version 7.0`. See the related Gotchas-section bullet about `Set-Content -Encoding utf8` BOM divergence between PS 5.1 and PS 7+.

## Docs

Specs and plans are kept side-by-side under `docs/superpowers/{specs,plans}/`, named `<YYYY-MM-DD>-<slug>.md`. Per-version release bodies live under `docs/release-notes/v<X.Y.Z>.md`. The central backlog and resolved-items log lives at `BACKLOG.md` at the repo root.

- Current: v0.4 macOS port ŌĆö `specs/2026-04-26-v0.4-macos-design.md` + `plans/2026-04-27-v0.4-macos-implementation.md`. Prior version specs/plans live alongside in `docs/superpowers/` (find via `ls docs/superpowers/{specs,plans}/`).
- **Backlog conventions:** open items, planned work, and resolved-item history live in `BACKLOG.md`. New items append at the bottom (`## <N>. <title>`) with body, fix candidates, urgency, and a `Touches:` file list. Resolved items get an inline `**Status: RESOLVED <date> (PR #<N>).**` marker AND an entry in the bottom `## Status > Resolved` section. The Status section also maintains an at-a-glance Open list and Provenance notes (how items were sourced). Inline-marker convention started ~PR #24 and was retroactively backfilled for older items in PR #41.
- A `**Validated <date> (v<X.Y.Z>).**` paragraph in a backlog item's body is distinct from the `Status: RESOLVED` line: RESOLVED = code merged; Validated = user-observable effect confirmed in the wild (e.g. beta-tester pass against a specific tag). Useful for items that ship across alpha cycles where merge and validation happen in different versions. Item #33 is the canonical example.
- When filing-and-resolving a backlog item in the same PR, predict the PR # for the inline `Status: RESOLVED ... (PR #<N>).` marker via `gh pr list --state all --limit 1` plus `gh issue list --state all --limit 1` (next available is `max(numbers) + 1`). Project's low parallel-PR activity makes prediction reliable; verify after `gh pr create` and amend the marker if wrong (rare, but happens if multiple PRs/issues open simultaneously).
- A tracker item's prescribed fix can be at the wrong layer ŌĆö verify against current code/docs before executing. Twice-validated: item 21 framed the fix as `_backends/windows_ads.is_ignored` but the right layer was `reconcile._reconcile_path` (covers Linux ENOTSUP-on-read too); item 22 prescribed "rewrite the paragraph" but the README's top-level `## Upgrading from v0.2.x` section already had the correct content, so the fix was deletion. Read the surrounding code or docs before assuming the prescribed fix is right; record the framing correction in the inline RESOLVED marker so future readers see it.

## How to run checks

Use these commands before claiming a change is safe:

```bash
uv run mypy .
uv run ruff check . --fix
uv run ruff check .
uv run ruff format .
uv run python -m pytest
```

If a tool is not installed, say so and continue with the available checks.
