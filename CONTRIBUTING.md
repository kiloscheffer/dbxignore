# Contributing to dbxignore

Cross-platform Python utility: keeps Dropbox ignore markers (NTFS alternate data streams on Windows; `user.com.dropbox.ignored` xattrs on Linux; `com.dropbox.ignored` xattrs on macOS) in sync with hierarchical `.dropboxignore` files.

## Commands

- `uv sync --all-extras` — install
- `uv run python -m pytest` — full suite (canonical form, also listed under "How to run checks"). Plain `uv run pytest` works on a fresh `uv sync` but can fail with `ModuleNotFoundError` or `uv trampoline failed to canonicalize` on stale environments — see Gotchas. Windows adds a few ADS-integration tests via `@pytest.mark.windows_only`.
- `uv run python -m pytest -m "not windows_only"` — portable subset; CI's cross-platform leg uses a broader exclusion (`"not windows_only and not linux_only and not macos_only"`) and each platform leg then runs its own `<platform>_only` step
- `uv run python -m pytest -W error::DeprecationWarning` — local strict mode (not enforced in CI)
- `uv run ruff check` — lint; rule families per `pyproject.toml` `[tool.ruff.lint] select` (don't restate the list here — pyproject is the source of truth); line length 100
- `dbxignore <apply|status|clear|list|explain|check-ignore|daemon|install|uninstall|init|generate>` — CLI console script (`cli:main`). `install` / `uninstall` register / remove the daemon with the platform's user-scoped service manager (Task Scheduler on Windows, systemd user unit on Linux, launchd LaunchAgent on macOS). `uninstall --purge` also clears every ignore marker.
- `dbxignore daemon` — starts the daemon; the platform's installed service (Scheduled Task / systemd unit / LaunchAgent) invokes the `dbxignore` console script with the `daemon` subcommand.
- `python -m dbxignore <subcommand>` — equivalent to the console script, via `src/dbxignore/__main__.py`. Useful when the wheel isn't installed (e.g. `uv run python -m dbxignore status`).

## Architecture

`reconcile.reconcile_subtree(root, subdir, cache)` is the single source of truth for rule-driven marker mutations. `cli.apply`, `daemon._dispatch`, and `daemon._sweep_once` all call it — never bypass. The lone exception is `cli.uninstall --purge`, which issues an unconditional marker clear (no rule evaluation) while still honoring the `.dropboxignore`-found-marked `WARNING` contract inline. After the marker clear, `--purge` also calls `_purge_local_state()` (removes `state.json`, `state.json.tmp`, `daemon.lock`, `daemon.log*`, `_test_slow_sweep`, the per-user state dir if empty) and on Linux additionally calls `linux_systemd.remove_dropin_directory()` to drop any user customization unit drop-ins. Goal: leave no dbxignore-authored artifacts on disk.

Marker I/O is platform-dispatched via `dbxignore.markers`, which at import time re-exports `is_ignored`/`set_ignored`/`clear_ignored` from `_backends/windows_ads.py` (Windows NTFS ADS), `_backends/linux_xattr.py` (Linux `user.com.dropbox.ignored`), or `_backends/macos_xattr.py` (macOS — see the module docstring for File Provider-mode detection logic and dual-attribute behavior). No other module branches on `sys.platform` for markers. `reconcile._reconcile_path` has symmetric error arms: both **read** and **write** sides catch broad `OSError`, log `WARNING` + append to `Report.errors`, and return a value that lets `reconcile_subtree` continue. Read returns `None` (vanished/unreadable path — don't drive subtree pruning); write returns `currently_ignored` (last-known marker state drives subtree pruning when an already-marked directory's write fails; returning `None` would re-walk into a subtree the filesystem can't mark and spam `WARNING`s per child). The write arm preserves a specific log message for `ENOTSUP/EOPNOTSUPP` ("Filesystem does not support ignore markers...") because that's a sysadmin-actionable distinction; other errnos log a generic `errno=NN` line. Real bugs (non-`OSError` exceptions) still propagate.

`daemon._sweep_once` fans `reconcile_subtree` out across roots via `ThreadPoolExecutor` (one worker per root). Safe because reconcile reads the cache lock-free (each `.get()` is GIL-atomic) and writes per-file ignore markers on disjoint paths. Multi-step traversals like `_applicable` are *not* transactional — between `.get()` calls a debouncer-thread mutation can change which ancestor's rules apply, so a single sweep tick may match against a slightly-stale ancestor view. Downstream is convergent: the next watchdog event or hourly sweep recovers, so the design accepts the drift instead of paying the per-`_applicable` lock cost. `RuleCache._rules` is guarded by a `threading.RLock` — any mutation (`load_root`, `reload_file`, `remove_file`, or the stale-purge iteration in `load_root`) must go through it, otherwise the debouncer thread can race with the main-thread sweep. If you add cross-root shared state to `RuleCache` or reconcile, revisit this.

`rules.RuleCache` stores one `_LoadedRules(lines, entries, mtime_ns, size)` per `.dropboxignore`. `entries` is a list of `(source_line_index, pathspec.Pattern)` pairs and is the single source of truth for both `match()` and `explain()`.

The static conflict detector lives in `rules_conflicts.py`; `RuleCache._recompute_conflicts` is its sole production caller. Public symbols (`Conflict`, `_detect_conflicts`) are re-exported from `rules.py` so callers can import from either module.

`rules._load_if_changed` skips reparse when a `.dropboxignore`'s `mtime_ns` and `size` both match the cached values — that's why `_LoadedRules` carries stat fields. The sweep path (`load_root`) uses it; watchdog-driven `reload_file` bypasses it because an explicit event is authoritative.

The daemon's watchdog events are classified (`_classify` → `EventKind.{RULES,DIR_CREATE,OTHER}`) and funneled through `Debouncer` before `_dispatch` runs `reconcile_subtree`. `DEFAULT_TIMEOUTS_MS` per kind is overridable via `DBXIGNORE_DEBOUNCE_{RULES,DIRS,OTHER}_MS`. One deviation from the funnel: `_WatchdogHandler.on_any_event` fast-paths a matched `DIR_CREATE` (the directory's path already matches a cached rule) by calling `reconcile_subtree` synchronously and skipping `Debouncer.submit` — this narrows the race window where Dropbox sees the new directory before its marker lands. Other event kinds (RULES, OTHER) and unmatched DIR_CREATEs still go through the debouncer. Trade-off: a queued-but-unprocessed RULES event that would invalidate the match means the bypass marks a path the next `reconcile_subtree` (driven by that RULES event) clears — bounded transient false-positive. On daemon startup, the observer + early `state.json` write happen *before* the initial sweep — the sweep itself runs in a worker thread (`_initial_sweep_worker`), so events arriving during the ~50s initial-sweep window on large trees are dispatched normally rather than dropped. Worker failure logs the traceback and sets `stop_event` to trigger the same shutdown path SIGTERM uses.

`install/` is a package with `__init__.py` exposing `install_service()` / `uninstall_service()` dispatchers, plus `windows_task.py` (schtasks XML generation + invocation), `linux_systemd.py` (writes `~/.config/systemd/user/dbxignore.service`, runs `systemctl --user daemon-reload && enable --now`), and `macos_launchd.py` (writes `~/Library/LaunchAgents/com.kiloscheffer.dbxignore.plist`, bootstraps via `launchctl bootstrap gui/<uid>`). `cli.install` / `cli.uninstall` delegate only to the package dispatcher; don't call backend modules directly from the CLI layer.

On Windows, `install/windows_shell.py` additionally writes two HKCU shell-integration verb keys (`HKCU\Software\Classes\AllFilesystemObjects\shell\DbxignoreIgnore` and `…\DbxignoreRestore`), opt-out via `dbxignore install --no-shell-integration`. The `Ignore from Dropbox` command runs without `--yes` (console-confirm because ignoring is destructive — Dropbox deletes from cloud and propagates), the `Restore to Dropbox` command runs with `--yes` (one-click safe). The `AppliesTo` filter is generated from `roots.discover()` and uses `:=` for the root itself plus `:~<` with a trailing `\` for descendants — together avoiding false matches on sibling folders like `Dropbox-other`.

`_windows_dialogs.should_use_gui_dialogs()` uses a two-stage probe: `GetConsoleWindow() == 0` AND `sys.stdout.fileno()` raises. Both checks are load-bearing: the old `sys.stdout is None` probe missed `dbxignorew.exe` (PyInstaller's noconsole bootloader leaves a stub writer, not None); the `GetConsoleWindow`-only probe misroutes `CREATE_NO_WINDOW` automation invocations to MessageBox even though `sys.stdout` is wired to a valid inherited pipe. `__main__._handle_explorer_double_click` pre-CLI hook pops a help MessageBox via `_windows_dialogs.show_info` when both conditions are met AND `len(sys.argv) <= 1` — the bare-double-click path that would otherwise exit silently after click writes its usage to a discarded stream.

The daemon prevents concurrent instances via `daemon._acquire_singleton_lock` — opens `daemon.lock` in the per-user state dir and holds an `fcntl.flock` (POSIX) / `msvcrt.locking` (Windows) for the daemon's entire lifetime. A second `dbxignore daemon` invocation gets `BlockingIOError` / `OSError` on the lock attempt and exits. The lock is the authoritative "is another daemon running?" gate.

`state.is_daemon_alive(pid: int | None, create_time: float | None = None) -> bool` is the canonical liveness check used by external observers (not the daemon itself) — verifies BOTH that the PID exists AND that the process at that PID is plausibly a dbxignore daemon (process name contains `"python"` or equals `"dbxignore"`, `"dbxignore.exe"`, `"dbxignorew"`, or `"dbxignorew.exe"`; without the name guard, a recycled PID claimed by an unrelated process registers as alive). When `create_time` is supplied, it additionally compares against `psutil.Process.create_time()` to reject PID-reuse races. `cli.status` and `cli.clear`'s daemon-alive guard delegate to it; new code touching daemon liveness should too.

Destructive CLI verbs that race the daemon's reconcile loop (today: `cli.clear`) refuse-to-run when `state.is_daemon_alive(s.daemon_pid)` is True — daemon reconcile reapplies markers on a sub-minute cadence and the hourly sweep reapplies wholesale, so a client-side mutation gets silently undone. `--force` overrides the guard for short-window tests; `--yes` skips the confirmation prompt for scripted use; `--dry-run` previews. Both `clear` and `apply` carry `--yes`/confirmation ceremony because both are destructive: `clear` causes Dropbox to upload previously-ignored paths (potentially gigabytes for a `node_modules`-class subtree); `apply` (and any rule edit that flows through the daemon) causes Dropbox to delete previously-synced paths from the cloud and propagate the deletion to every linked device — the marker is authoritative, leaving the local copy as the only surviving copy until cleared. The daemon-alive guard applies only to `clear` — the daemon performs `apply`'s operation continuously, so racing it is normal usage.

CLI fatal-error exits use `sys.exit(2)` project-wide. The verdict-driven commands `explain` / `check-ignore` use `0` (ignored) / `1` (not ignored) for the verdict surface AND `2` for fatal (no Dropbox roots, etc.) — preserving the project convention rather than git's `128`. New CLI commands should follow the `2`-for-fatal pattern.

`cli._normalize_under_root(path, *, require_exists)` is the thin core for CLI path validation (non-following normalize + Dropbox-root containment + optional `lexists`). `cli._validate_target_under_root(path)` wraps it adding `require_exists=True` plus symlinked-ancestor rejection. Filesystem-state verbs (`apply`, `clear`, `list`, `ignore`, `unignore`) use the full validator; rule-logic verbs (`explain`, `check-ignore`) use the thin core with `require_exists=False`. New path-taking verbs follow this split based on whether they touch filesystem state or only consult rules.

## Gotchas

- Rules/pathspec: subclass `GitIgnoreSpecPattern`; directory-only rules need a trailing `/` in tested strings; leading whitespace before `#` is an active pattern, not a comment; pathspec accepts many malformed-looking patterns, so tests for invalid-pattern handling should monkeypatch `rules._build_spec` to raise.
- CLI/rich-click: keep `cli.py`'s `from dbxignore import rules` module import so monkeypatches reach `rules._build_spec`; escape literal backticks in command docstrings because `rich_click.TEXT_MARKUP = "markdown"` strips unescaped ones; click 8.3+ exposes `Result.stdout` / `.stderr` directly.
- Marker invariants: `_backends/windows_ads` needs the `\\?\` ADS path prefix; NTFS matching is case-insensitive; `.dropboxignore` files are never ignored and must be cleared with a `WARNING` if found marked; `rules.match/explain` and marker APIs require absolute paths.
- Hot-path filesystem work: resolve paths at CLI/daemon boundaries, not inside cache/markers loops; keep `daemon._resolve_under_roots`'s cheap containing-root check before `Path.resolve()`; hoist repeated `is_dir()` calls as in `_rel_path_str(..., is_dir)`.
- Daemon state: `state.write()` must stay temp-file-plus-`os.replace`; `state.json` means the daemon started, not that initial sweep completed (`last_sweep is None` maps to `state=starting`); `_initial_sweep_worker` helper calls must remain inside its broad `try/except`.
- User-controlled wait durations: validate with `math.isfinite(value)` and `value <= threading.TIMEOUT_MAX` before `threading.Event.wait(timeout)`. `float("inf")` and `float("nan")` parse successfully.
- Symlinks are leaves. Linux refuses `user.*` xattrs on symlinks, macOS marks links via NOFOLLOW, and Windows ADS attaches to the reparse point. Tests must branch by platform; walk roots that are symlinks need explicit short-circuits because `os.walk(..., followlinks=False)` still follows the root itself.
- Rules conflicts: `RuleCache` recomputes static conflicts on every mutation. `match()` and reconcile ignore dropped entries; `explain()` returns them with `Match.is_dropped=True`.
- Logging/tests: `daemon._configured_logging()` snapshots and restores the `dbxignore` logger; Linux installs file + stderr handlers while Windows only installs the file handler. Tests that count handlers branch on `sys.platform`, and tests that trigger state writes monkeypatch `state.default_path`.
- Platform test shape: shared fixtures live in `tests/conftest.py`; platform-only modules set module-level `pytestmark` and skip at collection time on other platforms; status/clear daemon-alive tests monkeypatch `state.is_daemon_alive` instead of relying on the current pytest process name.
- Windows MessageBox-routing tests rely on `tests/conftest.py:_stub_get_console_window` (autouse, returns `0x1` = "console attached") because pytest itself has no console under `uv run python -m pytest` — without the stub, `_confirm_or_messagebox` / `_error_or_messagebox` / `_handle_explorer_double_click` would fire real `MessageBoxW` calls that block the runner. Per-test override via `monkeypatch.setattr(_windows_dialogs, "should_use_gui_dialogs", lambda: ...)` (preferred) or `monkeypatch.setattr(ctypes, "windll", FakeWindll(...))` (lower-level).
- Install/runtime packaging: `roots.discover()` honors `DBXIGNORE_ROOT` before Dropbox `info.json`; Windows checks both `%APPDATA%` and `%LOCALAPPDATA%`; packaged non-.py resources live under `src/dbxignore/**` and should be read with `importlib.resources`.
- macOS/Linux install details: macOS xattr sync-mode behavior is documented in `src/dbxignore/_backends/macos_xattr.py`; macOS launchd uses modern `launchctl bootstrap` / `bootout`; Linux systemd forwards path-resolution env vars and must double `%` before C-style env escaping.
- `AppliesTo` AQS string-literal escape: AQS does NO escape interpretation inside quoted string literals — backslashes are stored and matched literally. A single trailing `\` immediately before the closing `"` is parsed as a literal trailing backslash without escaping the quote. So a path `C:\Foo` is stored as `"C:\Foo"`, and the prefix-match form needs `"C:\Foo\"` (single trailing `\`) to disambiguate from sibling folders like `C:\Foo-other`. `.reg` file syntax adds its own escape layer on top — `.reg` text `"C:\\Foo"` represents the same stored REG_SZ value as the AQS query string `"C:\Foo"`, easy to confuse when comparing examples. `str(Path("D:/"))` on Windows already ends in `\`, so prefix-clause construction must `.rstrip("\\") + "\\"` to avoid double-trailing (see `_format_applies_to_query` in `install/windows_shell.py`).
- `winreg.DeleteKey` deletes only leaf keys: walk subkeys bottom-up (child → parent). Failures cascade — a failed child-delete blocks the parent-delete with ERROR_ACCESS_DENIED, so error accumulators may collect >1 entry per intended deletion (see `tests/test_install_windows_shell.py::test_uninstall_other_oserror_routes_to_errors_or_warning` for the `>= 1` relaxation this forces).
- PyInstaller specs: `from PyInstaller.utils.hooks import copy_metadata` + `datas=copy_metadata("dbxignore")` in `Analysis(...)` is required for `@click.version_option(package_name="dbxignore")` to work inside the frozen bundle — without it `--version` raises `RuntimeError: 'dbxignore' is not installed` (stderr traceback on `console=True`; GUI traceback dialog on `console=False`). CI smoke-test in `.github/workflows/release.yml` exercises both `--help` AND `--version` to catch this regression class — `--help` short-circuits before the version callback fires.
- Windows GUI-subsystem (`console=False`) foreground processes: PowerShell does NOT wait synchronously — the prompt redraws before the binary's stdio flushes, so output races the prompt and looks lost. AttachConsole + per-stream stdio rehydration in the child cannot fix this (the shell's wait behavior is decided by the PE subsystem byte at link time; cmd.exe waits, PowerShell doesn't). The project ships a dual binary (`dbxignore.exe` console=True + `dbxignorew.exe` console=False, mirroring `python.exe`/`pythonw.exe`) rather than a single AttachConsole binary. Don't try collapsing to one binary again without a different strategy.
- Testing lazy `from x.y import z` imports on other platforms: `monkeypatch.setattr(pkg, "y", fake)` silently misses (the attribute doesn't exist yet because the lazy import hasn't fired); use `monkeypatch.setitem(sys.modules, "x.y", fake_module)` instead so the lazy import resolves through `sys.modules`. See `tests/test_install.py::_inject_fake_windows_shell` for the pattern.
- Local environment traps: prefer `uv run python -m pytest` over plain `uv run pytest`; Windows cloud-synced dirs can require `UV_LINK_MODE=copy`; reinstalling the uv tool while the Windows daemon is running fails until `dbxignore uninstall` releases mapped files; `uv tool install <local-path>` caches built wheels under `sdists-v9/path/<dir-hash>/` keyed on source-dir path NOT git SHA — commits don't invalidate the cache. `uv cache clean dbxignore` before reinstall when testing local commits (manual-test scripts auto-clean in Phase 2).
- Type/check tooling: cross-platform `# type: ignore` codes often need combined forms such as `[import-not-found, import-untyped, unused-ignore]`; keep critical test deps mirrored in both `[project.optional-dependencies].dev` and `[dependency-groups].dev`.
- Timing/debugging: use `_logging.timed_debug(...)` for hot-path timing; `pytest-timeout` default is 10s, so longer poll/shutdown tests need `@pytest.mark.timeout(N)`.
- Failure injection for manual tests: `src/dbxignore/_testing.py` is the home of the `DBXIGNORE_TEST_FAIL_*` env-var failure-injection convention. `raise_if_fail_point(name)` (raises `OSError` into an existing `except OSError` arm) and `fail_point_active(name)` (predicate for value-substitution sites) let the manual-test shell scripts drive exit-2 error paths whose underlying failure mode can't be forced on a healthy machine. Current fail points: `MARKER_READ`, `STATE_PURGE`, `BOOTOUT`, `DAEMON_ALIVE` (see the module docstring). A new "hard-to-force exit-2 path" is a one-line hook at the boundary plus a docstring entry — don't re-defer it as untested manual-test surface.
- Deeper gotchas live in `docs/internals/active-gotchas.md`; long-form design-rationale and rarely-needed gotchas live in `docs/internals/long-form-gotchas.md`. Read the relevant internals doc before touching daemon startup, installers, platform marker backends, commit-check workflows, or case/symlink traversal behavior.

## Git workflow

- Never commit directly to `main`. Work on a topic branch and open a PR — that's what triggers `.github/workflows/test.yml` (the platform-gated test tiers `pytest -m windows_only`, `pytest -m linux_only`, and `pytest -m macos_only` **only run in CI**) and `.github/workflows/commit-check.yml` (commit-message + branch-name validation). A local `uv run pytest` can only exercise one platform, so a single green local run is not a merge gate. The PR matrix is.
- **`cchk.toml` at repo root is the single source of truth** for allowed commit types (Conventional Commits, see `allow_commit_types`), branch types (Conventional Branch, see `allow_branch_types`), and the subject-length cap. Don't restate those lists here or elsewhere — reference the file so it can't drift. Local enforcement is optional but encouraged: `uv tool install pre-commit && pre-commit install --hook-type commit-msg --hook-type pre-push` wires the same rules at commit/push time. CI re-runs them on every PR via `commit-check/commit-check-action@v2.6.0`.
- **Pre-flight commit-check against every commit, not just HEAD.** CI runs `commit-check` over the full `origin/main..HEAD` range — a check that only validates the planned PR title can pass locally while an intermediate commit fails CI (e.g. a commit description starting with `--` trips commit-check's regex). Loop over each subject before pushing.
- **Commit subjects starting with `#` immediately after the type/scope colon are rejected by commit-check** as if the whole message were a comment. `docs(backlog): #<N> sixth observation` fails; `docs(backlog): log item #<N> sixth observation` passes. Anchor the description with a non-`#` token first when the natural phrasing would put a backlog reference at the start.
- Branch names follow `<type>/<slug>` where `<type>` is from `cchk.toml`'s `allow_branch_types` and `<slug>` is lowercase-alphanumeric + hyphens. Note two asymmetries with commit subjects: (1) the branch prefix `feature/` is the long form while the Conventional Commits subject tag `feat:` is the short form; (2) `allow_branch_types` is a **strict subset** of `allow_commit_types` — `docs/`, `style/`, `refactor/`, `perf/`, `test/`, `build/`, `ci/`, `revert/` are valid commit types but NOT valid branch prefixes. Use `chore/` for those categories of work. Examples: `feature/linux-backend`, `fix/xdg-paths`, `chore/release-prep`.
- Commit subjects follow Conventional Commits: `<type>(<scope>): <description>` where `<type>` is from `cchk.toml`'s `allow_commit_types`. Scope tags mirror package names or doc categories, not ticket numbers. `!` before the colon — or a `BREAKING CHANGE:` footer — signals a breaking change.
- **commit-check rejects multi-element scopes** (comma-separated): `fix(rules,daemon): ...` fails. Use a single scope (the most-affected module) or omit the scope.
- Split commits along revertability lines: a code change and a doc-only backlog update belong in separate commits because they could plausibly be reverted at different times — one `feat` commit for a behavior change, a separate `docs` commit for new follow-up entries.
- Avoid parallel PRs that each edit BACKLOG.md's bottom Open list — particularly when both add new items at the end or both touch adjacent bullets. Sequential merges leave the second-and-later PRs with mechanical conflicts on the Open-list block. Three options: stack PRs (each branches off the previous one's branch), serialize merges (wait for each before opening the next), or defer BACKLOG bookkeeping to a single combined PR at session end.
- Items filed and resolved in the same PR don't need to land in `BACKLOG.md` — the PR description is sufficient context. If a `<THIS_PR>` placeholder is left in source/test docstrings or manual-test scripts to reference the eventual PR number, fill it in via a follow-up `docs: fill in PR #<N>` commit after `gh pr create` returns the number; the fill-in commit must `grep` the **whole repo** for `<THIS_PR>` to catch every site (the placeholder can land in multiple files).
- Run any post-implementation cleanup (lint, format, small refactors) **before** requesting merge. A commit pushed to a branch whose PR already merged is stranded — recover by cherry-picking it onto a fresh branch off updated `main` (the merged PR guarantees the orphan's parent tree is in `main`, so the cherry-pick is conflict-free).

## Release

- `.github/workflows/test.yml` runs ruff + the portable pytest subset on `ubuntu-latest`, `windows-latest`, and `macos-latest` for every PR (via `pull_request` events) and every push to `main` (catches merge commits + any direct-push escape). Topic-branch pushes without an open PR do NOT trigger CI. Each platform leg additionally runs its own `pytest -m <platform>_only` step. A `concurrency` group keyed on `${{ github.workflow }}-${{ github.ref }}` with `cancel-in-progress: true` cancels superseded runs on the same ref — saves Windows-runner minutes (10× cost multiplier vs Linux) during fast PR iteration.
- Push tag `v*` → `.github/workflows/release.yml` builds wheel + the Windows onedir bundle archived as `dbxignore-windows-x86_64.zip` (via `pyinstaller/dbxignore.spec`) + the Windows Inno Setup installer `dbxignore-setup.exe` + `dbxignore-macos-arm64.tar.gz` (macOS arm64 onedir bundle, via `pyinstaller/dbxignore-macos.spec`) + `dbxignore-linux-x86_64.tar.gz` (Linux x86_64 onedir bundle, via `pyinstaller/dbxignore-linux.spec`) and publishes to two destinations: GitHub Release (auto, tag-gated; pre-release flag sourced from the `classify-tag` job) and PyPI (only wheel/sdist ship to PyPI — the `.zip` bundle, `dbxignore-setup.exe`, and the macOS and Linux tarballs are GitHub-Release-only). `hatch-vcs` derives the version from the tag — no manual `pyproject.toml` bump needed. PyPI uses Trusted Publishing; no API token secret stored.
- A `classify-tag` job is the single source of truth for "is this a pre-release tag?" — it parses the tag in shell against PEP 440 patterns (`aN`, `bN`, `rcN`, `.devN`) and emits an `is_prerelease` boolean output. `publish-github`'s `prerelease:` flag and `publish-pypi`'s `if:` clause both read from that output, so they cannot disagree. **Do NOT use `contains(github.ref, '-a')`-style checks** — PEP 440 has no hyphen separator (alphas are `0.4.0a1`, not `0.4.0-a1`), so substring checks against hyphenated SemVer-style markers always evaluate False against the tags `hatch-vcs` produces.
- PyPI publish has two safety layers: (1) PEP 440 pre-release tags skip the `publish-pypi` job entirely via the `classify-tag`-driven `if:` clause — they're never queued, so there's nothing to accidentally approve; (2) release tags enter the job and pause at the `pypi` GitHub environment gate, which requires a maintainer approval click in the GitHub UI before the OIDC upload fires. The two layers cover different threat models: the `if:` exclusion prevents pre-releases from hitting PyPI even if someone clicks approve; the environment gate prevents release tags from auto-publishing without human review. To pre-release without PyPI: tag with PEP 440 pre-release suffix (`v<X.Y.Z>a<N>`, `v<X.Y.Z>rc<N>`, `v<X.Y.Z>.dev<N>`). To release: tag `v<X.Y.Z>` and approve at the gate. Note: `.postN` (post-release) is intentionally NOT a pre-release — post-release tags publish to PyPI.
- Before tagging, sanity-check wheel metadata: `uv build && unzip -p dist/*.whl '*.dist-info/METADATA' | head` — confirms `Name: dbxignore` and the expected `Version:` are in the wheel before a misnamed upload hits PyPI (immutable; yank-only recovery).
- This project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html). Pre-1.0, breaking changes ride MINOR bumps. Post-1.0, breaking changes will bump MAJOR. Conventional Commits' `!`-or-`BREAKING CHANGE:` syntax (see Git workflow above) signals breakage either way.
- Watch a long-running `release.yml` run via `gh run watch <run-id> --exit-status`. Don't poll via shell sleep loops — burn rate climbs without bound and signal is low.
- Third-party GitHub Actions in `.github/workflows/*.yml` are pinned to **40-char commit SHAs** with a trailing `# v<X>` comment naming the resolved tag — `uses: actions/checkout@93cb6efe18208431cddfb8368fd83d5badbf9bfd # v5`. Tags are mutable (the action's owner can re-point them at any commit); SHAs are content-addressed. Resolve a tag's SHA via `gh api repos/<owner>/<repo>/commits/<tag> --jq .sha`. `.github/dependabot.yml` runs `package-ecosystem: github-actions` weekly with `commit-message.prefix: "ci"` + `include: "scope"` so auto-PRs land as `ci(deps): bump …` and pass `cchk.toml`'s Conventional Commits gate; minor+patch bumps are grouped into one weekly PR via `actions-minor-patch`, major bumps stay individual for review. New workflows MUST follow the SHA-pin convention from day one — a tag-pinned action in a new file silently degrades the trust posture across the project.
- Workflow `if:` filters that gate on "is this a bot's PR?" must use **`github.event.pull_request.user.login`**, not `github.actor`. `github.actor` is whoever triggered the *current* run — when a human clicks "Update branch" on a Dependabot PR, `github.actor` becomes the human and a `!endsWith(github.actor, '[bot]')` filter no longer fires, so the gated step runs anyway. `pull_request.user.login` is the PR *author* and stays `dependabot[bot]` across re-runs. Different intent than "did a bot push the latest commit?" (which `claude-code-review.yml:25` correctly answers via `github.actor`); use the field that matches the question.
- **`workflows` is NOT a valid `GITHUB_TOKEN` permissions scope.** GitHub refuses any `git push` from the default `GITHUB_TOKEN` that modifies files under `.github/workflows/**`, regardless of what's in the workflow's `permissions:` block. Workflow-file pushes need either a PAT with `workflow` scope or a GitHub App installation token with `workflows` write set at the App level.

## Manual test scripts

`scripts/manual-test-{ubuntu-vps,macos,windows}.{sh,ps1}` are end-to-end smoke tests requiring a live Dropbox install. CI does not run them — they're for release-prep verification of behavior the unit tests can't reach (real markers on a real synced tree, real service-manager registration, real cross-process daemon interaction). All three follow the same phase numbering: 0 (preflight), 1 (verify Dropbox), 2 (install dbxignore), 3 (CLI surface), 4a–4f (reconcile), 4.5 / 4g–4n (extended CLI), 5 (daemon), 6 (uninstall), 7 (cleanup).

- **The manual-test scripts are kept current** — extending them is a **hard requirement** for every change that touches user-visible CLI surface, not an optional nicety. Triggers: new subcommand, new flag with marker/Dropbox side effects (`--yes`, `--dry-run`, `--purge`, `--force`), new stderr warning, new exit code, or behavior change unit tests can't fully cover. The change that introduces the surface MUST land its Phase 4.5 (or, for uninstall/daemon-coupled surface, Phase 5/6) extension alongside it — inline extension is far cheaper than a later backfill, which has to reconstruct the surface's context from scratch. Backfilling later is allowed only when the inline extension is impractical (e.g. forcing a real OSError needs platform-specific setup), and the limitation has to be called out. The "always file follow-ups" rule below applies: if the case can't be added inline, file a backlog item to cover it. Each case carries an inline `# 4X — <description>` comment.
- **Phase 4.5 layout.** The two bash scripts source `scripts/_phase_extended_cli.sh` for the shared Phase 4.5 body; the PowerShell script can't share the helper and is hand-synced. New Phase 4.5 cases land in the bash helper plus `manual-test-windows.ps1`'s `Test-ExtendedCli`.
- **Cross-platform behavioral divergences** branch per-script with inline comments. Canonical examples: 4e symlinks (Linux refuses `user.*` xattrs with EPERM → WARNING; macOS allows via the NOFOLLOW path; Windows attaches ADS to the reparse point → marks silently). Don't paper over these — the manual scripts are where the divergences are surfaced for testers, and each platform script inverts the assertion accordingly.
- **PowerShell scripts** (`scripts/manual-test-windows.ps1`) start with `#requires -Version 7.0`. See the related Gotchas-section bullet about `Set-Content -Encoding utf8` BOM divergence between PS 5.1 and PS 7+.

## Docs

Active gotchas live under `docs/internals/active-gotchas.md`; long-form gotchas and design-rationale notes live under `docs/internals/long-form-gotchas.md`. The central backlog lives at `BACKLOG.md` at the repo root.

- **Backlog conventions:** open items and planned work live in `BACKLOG.md`. New items append at the bottom (`## <N>. <title>`) with body, fix candidates, urgency, and a `Touches:` file list, plus a one-line bullet under `## Status > Open` referencing the new number. When an item ships, remove its `## <N>.` entry AND its `Status > Open` bullet — leave a gap in the numbering rather than renumbering the rest (renumbering orphans every PR/source/script reference to the old number). The shipped item's history lives in the merging PR's description and the commit log. Items that are merged but awaiting field validation can stay open with a `**Merged <date> (PR #<N>); awaiting validation.**` paragraph in the body and get removed once validation lands.
- **Always file follow-ups and findings as you go.** When PR review, code work, or an operational task surfaces a related code issue, a design-tension observation, or a failure mode worth remembering, file it before the work-context evaporates. Concrete code items with at least one fix candidate (or an explicit "no fix yet, awaiting trigger" rationale, like the rule-of-three design-tension records) go in `BACKLOG.md` (numbered, with body, fix candidates, urgency, `Touches:`). Preventive development-discipline observations or implementation-pattern gotchas go in `docs/internals/active-gotchas.md` under the matching subsection — or directly in this file's Gotchas section above if the observation is short and broadly applicable enough for the top-level contributor doc. Underbaked filed entries are cheap; forgotten findings are expensive.

## How to run checks

Use these commands before claiming a change is safe:

```bash
uv run mypy .
uv run ruff check . --fix
uv run ruff check .
uv run ruff format .
uv run python -m pytest
```

If a tool is not installed, say so and continue with the available checks.
