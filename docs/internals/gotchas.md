## Gotchas

- pathspec 1.0.4: subclass `GitIgnoreSpecPattern`, not deprecated `GitWildMatchPattern`.
- pathspec: directory-only rules (`node_modules/`) require trailing `/` on the tested path string.
- pathspec follows gitignore's column-0 comment rule: a line with leading whitespace before `#` (e.g. `"   #literal"`) is an *active pattern*, not a comment. `rules._build_entries`'s filter checks `raw.startswith("#")` (not `raw.strip().startswith("#")`) so the line is correctly classified as active. The count-mismatch fallback at the bottom of `_build_entries` is now defensive scaffolding for future pathspec-version drift; under the gitignore-correct filter, fast-path counts match in practice.
- pathspec is liberal: gitwildmatch accepts `[invalid`, `**/*.{a,b`, etc. without raising. To test code that catches `(ValueError, TypeError, re.error)` from `_build_spec`, monkeypatch `rules._build_spec` to raise rather than searching for an input that triggers it.
- `cli.py` deliberately uses `from dbxignore import rules` (module attribute access) so `cli.generate` and `cli._apply_from_gitignore` call `rules._build_spec(...)` via lookup at use time. Tests rely on this: `monkeypatch.setattr(rules, "_build_spec", ...)` reaches the call site. Switching to `from dbxignore.rules import _build_spec` (symbol import) would silently break `tests/test_cli_generate.py::test_generate_invalid_pattern_writes_nothing`.
- click + rich_click: the project sets `rich_click.TEXT_MARKUP = "markdown"`, which strips unescaped backticks from command docstrings before they reach `--help` output. Escape them in the docstring source to preserve literal backticks for help-text assertions (e.g. `"Alias of \`explain\`"` in `tests/test_cli_status_list_explain.py`).
- click 8.3+ `Result` has separate `.stdout` / `.stderr` attributes by default; the older `CliRunner(mix_stderr=False)` API was deprecated. New CLI tests that need stderr assertions use `runner.invoke(...).stderr` directly (no kwarg needed). Project pinned at click 8.3.2 as of PR #195.
- `_backends/windows_ads` uses `open(r"\\?\path:com.dropbox.ignored")` directly — `\\?\` prefix mandatory for >260-char paths.
- NTFS is case-insensitive; `_CaseInsensitiveGitIgnorePattern` prepends `(?i)` to compiled regexes.
- `.dropboxignore` files are never marked ignored — guarded in `match()` and `explain()`; `reconcile._reconcile_path` clears any ADS marker it finds on one and logs at `WARNING` (spec contract — don't silence it in a refactor).
- `rules.match/explain` and `markers.{is,set,clear}_ignored` all require **absolute** paths and raise `ValueError` on relative ones. Resolve at the CLI/daemon boundary, never inside the cache or markers layer — `Path.resolve()` on Windows is a per-call syscall that dominated sweep wall-clock before.
- `daemon._resolve_under_roots(raw_path, roots)` is the canonical helper for the watchdog path: returns `(root, resolved_path)` if the raw path is under a watched root, else `None`. The order matters — `find_containing` runs *before* `Path.resolve()`, so events outside any root pay zero resolve cost. Used by `_classify` (`event.src_path`) and `_dispatch` (`event.dest_path` on `moved` events). Don't invert the order in a future refactor.
- `watchdog.observers.Observer` is a platform-conditional factory function, not a class — annotating a parameter as `Observer` fails mypy with `[valid-type]`. For type annotations, import `BaseObserver` from `watchdog.observers.api` and (since `from __future__ import annotations` is in effect, ruff TC002) place it under `if TYPE_CHECKING:`. The runtime construction `observer = Observer()` stays as today.
- `_rel_path_str(ancestor, path, is_dir)` takes `is_dir` as a parameter, not because the function needs it polymorphically but because `match()` / `explain()` compute `path.is_dir()` once per call and pass the bool down to every ancestor iteration — the directory-only-pattern check (`node_modules/`) was repeating the syscall D times for the same path when D ancestor `.dropboxignore` files applied. Same shape as the `Path.resolve()` rule above: stat syscalls in the per-file sweep loop are expensive in aggregate, hoist them to the per-call layer.
- `daemon._configured_logging()` is a context manager: it snapshots the `dbxignore` logger on enter and restores handlers/propagate/level on exit. On Linux it installs two handlers — `RotatingFileHandler(daemon.log)` plus `StreamHandler(sys.stderr)` so systemd-journald captures the same records — on Windows only the file handler.
- Test implications of the above: tests that count handlers must branch on `sys.platform`. `run()` wraps its body in `_configured_logging`, so tests calling `daemon.run()` don't need to hand-restore logger state — but if you mock the context manager out, use `contextlib.nullcontext` (see `test_daemon_singleton.py`).
- Test helpers (`FakeMarkers`, `fake_markers` fixture, `write_file` fixture) live in `tests/conftest.py` and are auto-available to every test module.
- Windows-only tests: set `pytestmark = pytest.mark.windows_only` at module level and guard with `if sys.platform != "win32": pytest.skip(..., allow_module_level=True)` so non-Windows collection skips cleanly. Same shape for `linux_only` and `macos_only`.
- Daemon/sweep tests that trigger state writes: `monkeypatch.setattr(state, "default_path", lambda: tmp_path / "state.json")` redirects the persisted state off the real per-user state dir and keeps the test hermetic.
- Per-user state dir varies: `%LOCALAPPDATA%\dbxignore\` on Windows, `$XDG_STATE_HOME/dbxignore/` (fallback `~/.local/state/dbxignore/`) on Linux, `~/Library/Application Support/dbxignore/` on macOS. Single source of truth is `state.user_state_dir()`. macOS additionally splits the daemon log into `~/Library/Logs/dbxignore/` per Apple's app-data conventions — `state.user_log_dir()` exposes this; on Windows + Linux it falls through to `user_state_dir()`. `daemon._log_dir()` reads from `user_log_dir()`. `cli._purge_local_state()` cleans both dirs on darwin.
- `state.write()` writes to `state.json.tmp` then `os.replace`s into place (POSIX-atomic on Linux; `MoveFileExW(MOVEFILE_REPLACE_EXISTING)` on Windows). Direct `path.write_text()` would truncate-then-write — a SIGKILL or power loss between the two leaves a zero-length / partial `state.json`; `_read_at` then returns `None` (`JSONDecodeError`) and `daemon.run`'s singleton check thinks no prior daemon exists, allowing two concurrent daemons. Apply the same temp+replace pattern for any future state-file writes; `_purge_local_state` cleans a leaked `state.json.tmp` if one is found.
- `state.json` existing means "daemon is alive" — NOT "daemon has completed an initial sweep." The early `state.write()` fires before the worker starts, so consumers checking `state.json` for readiness should also branch on `state_obj.last_sweep is None` to detect the `state=starting` window. `cli.status --summary` exposes this as a distinct `state=starting` token; callers parsing the summary need to handle the new value alongside `running`/`not_running`/`no_state`.
- Manual-test scripts can deterministically pad the daemon's initial sweep (so `state=starting` is observable on small trees) by writing a positive-number-of-seconds string into `state.user_state_dir() / "_test_slow_sweep"` before `dbxignore install`. `daemon._slow_sweep_pad_seconds()` reads it from `_initial_sweep_worker`, logs a WARNING when honored, and the `stop_event.wait(pad_s)` is interruptible so shutdown stays prompt mid-pad. Marker file rather than env var because (1) the install layer's `_FORWARDED_ENV_VARS` was scoped to a small allowlist, and (2) Windows Task Scheduler XML has no per-task env-var forwarding mechanism — a marker file works uniformly across all three platforms with zero install-layer surface change. `cli._purge_local_state` cleans it up on `uninstall --purge` so a stale marker can't silently pad future installs.
- `_initial_sweep_worker` runs in a worker thread with a `try/except Exception:` arm that logs and sets `stop_event` for clean daemon shutdown. **All new code in the worker (helper calls, env reads, validators) must live INSIDE this try block** — exceptions escaping the worker kill the thread silently and strand the daemon in `state=starting` indefinitely (the periodic-sweep guard at `daemon.py:717` keeps deferring to the worker that's never coming back). Failure modes that have actually bitten: `OverflowError` from `wait(inf)`, `UnicodeDecodeError` from non-UTF-8 marker contents. Test the contract: monkeypatch the new code to raise, assert `stop_event.is_set()` AND `_sweep_once` was not called.
- `threading.Event.wait(timeout)` raises `OverflowError` on non-finite values and on values exceeding `threading.TIMEOUT_MAX` (Windows ~4.3M sec, POSIX much larger — a Linux-picked test value can over-shoot Windows). User-derived wait durations need explicit `not math.isfinite(value) or value > threading.TIMEOUT_MAX → ignore` validation before calling `wait()`. `_slow_sweep_pad_seconds` does this for the slow-sweep marker; mirror the pattern for any future user-controlled wait duration. `float("inf")` and `float("nan")` parse successfully via `float(raw)` and slip past naive `value > 0` / `value < 0` arms — explicit `math.isfinite` rejection is the only catch.
- Root-discovery test seams, by layer: CLI tests monkeypatch `cli._discover_roots`; daemon tests monkeypatch `daemon.roots_module.discover`; `roots.discover()` unit tests use `_stage_info(monkeypatch, tmp_path, fixture_name)` which sets `APPDATA` (Windows) or `HOME` (Linux) to point at a staged `Dropbox/info.json` / `.dropbox/info.json`. Pick the layer that matches the code path under test.
- Linux xattrs vanish silently through common operations: `cp` without `-a`, cross-filesystem `mv`, most archivers, and `vim`'s default save-via-rename. The watchdog event stream + hourly sweep are the recovery mechanism — don't add a preservation wrapper, the design intentionally leans on reconcile.
- **Symlinks: platform-divergent and treated as leaves.** Marker-write behavior diverges by platform — Linux raises `PermissionError` on `set_ignored`/`clear_ignored` (kernel refuses `user.*` xattrs on symlinks; EPERM) and returns False from `is_ignored` (ENODATA); macOS marks the link itself via NOFOLLOW (`reconcile._reconcile_path`'s `PermissionError` arm is dormant); Windows ADS attaches to the symlink reparse point. Tests asserting symlink behavior must branch on `sys.platform`. Project rule for "is this a directory?" checks against user/rule paths: **symlinks are leaves.** Hot-path helper `rules._is_real_dir(path)` does a single `lstat()` + `S_ISDIR` (used in `cache.match:549`, `cache.explain:581`); cold paths use the two-call form `path.is_dir() and not path.is_symlink()` (`format_literal_rule:183`, `cli._matches_target_directly:321` — migration to the helper tracked as backlog #106). Walk-root `is_symlink()` short-circuit at three sites (`daemon._sweep_once` per-subdir fan-out, `cli._run_apply_pass` via `descend=False` to `reconcile_subtree`, `cli._walk_marked_paths`) because `os.walk(path, followlinks=False)` still follows when `path` itself is a symlink — `followlinks` only gates subdirectory traversal. Otherwise `ignore <link-to-dir>`'s no-slash rule fails to round-trip against `cache.match`'s trailing-slash lookup.
- `roots.discover()` branches on `sys.platform`: Windows checks `%APPDATA%\Dropbox\info.json` (per-user installer) AND `%LOCALAPPDATA%\Dropbox\info.json` (per-machine "install for all users") in priority order — the per-machine path was added when a v0.4 alpha tester reported "No Dropbox roots found" on a per-machine install; Linux and macOS use `~/.dropbox/info.json` (folded into one arm because Dropbox's desktop client uses the same path on both). Same JSON schema across all platforms. The `_info_json_paths()` helper returns a `list[Path]` of candidates (empty on unsupported platforms or when env vars are missing); `discover()` iterates and uses the first existing file, or warns and returns `[]` if none exist.
- `roots.discover()` honors `DBXIGNORE_ROOT` as a pre-`info.json` escape hatch for non-stock Dropbox installs. Set to an existing absolute path → `[Path(env)]`, bypassing `info.json` (and the platform check — useful on platforms where `_info_json_path()` returns `None`). Set to a nonexistent path → WARNING + `[]`. Empty string → treated as unset. Single-root only; multi-account setups with an override have to pick one.
- `RuleCache` runs a static conflict detector at every mutation (`load_root`, `reload_file`, `remove_file`). Negations whose literal path prefix lives under a directory matched by an earlier include rule are recorded in `_conflicts`, and their `(source, line_idx)` tuple is added to `_dropped`.
- Downstream consumers diverge: `match()` and reconcile ignore anything in `_dropped`; `explain()` still returns dropped matches but sets `Match.is_dropped=True` so CLI/log formatters can annotate them. Semantic reason: Dropbox's ignored-folder inheritance makes negations inert under ignored ancestors, so we don't pretend they're in effect.
- Local pytest gotcha: `uv run pytest` may not pick up the editable install after `uv pip install -e .`. Two failure modes seen: silent `ModuleNotFoundError: No module named 'dbxignore'`, and `error: uv trampoline failed to canonicalize script path`. Same workaround in both cases — use `uv run python -m pytest`, which invokes the interpreter that has the `.pth` file. CI doesn't hit this (fresh provisioning).
- Windows install gotcha: `uv tool install git+...` can fail with `ERROR_CLOUD_FILE_INCOMPATIBLE_HARDLINKS` (os error 396) when `%AppData%` is OneDrive-synced via Files On-Demand. uv hardlinks files from `%LOCALAPPDATA%\uv\cache\` into `%APPDATA%\uv\tools\<pkg>\Lib\site-packages\` by default; the Cloud Files API refuses hardlinks on placeholder files. Workaround documented in the README's Windows install section: `uv tool install --link-mode=copy git+...` (or `$env:UV_LINK_MODE = "copy"`). CI doesn't hit this — the GitHub Windows runner has no cloud-synced AppData.
- Windows + running daemon: `uv tool install --force --reinstall <wheel>` (or any reinstall against an already-installed `dbxignore` tool) fails with `os error 5` ("Access is denied") on `<uv-tools>\dbxignore\Lib` if the daemon process is up — the daemon holds `.pyd` files mapped. Run `dbxignore uninstall` first: it's synchronous (schtasks `/End` + wait for daemon-process exit + `/Delete`), so the file locks are released before it returns. Bare `schtasks /End /TN dbxignore` is NOT enough on its own — `/End` signals the task to stop but returns immediately, and the daemon can still hold the locks when the immediate retry fires. After the reinstall, `dbxignore install` re-registers the task and starts the new daemon.
- The `xattr` PyPI package's module-level `getxattr`/`setxattr`/`removexattr` use `symlink=True` to operate on the link itself (NOT `options=xattr.XATTR_NOFOLLOW` as the underlying macOS C API would suggest). The kwarg is named `symlink` and its boolean is "operate on the link, don't follow." Verified the hard way during PR 55's first macOS CI run — the v0.4 spec had projected from Apple's libc docs and was wrong. Anyone touching `_backends/macos_xattr.py` should match the package's actual signature, not the platform's.
- macOS daemon installer (`install/macos_launchd.py`) uses modern `launchctl bootstrap gui/<uid> <plist>` and `bootout gui/<uid>/<label>` — not legacy `load -w` / `unload -w`. Modern bootstrap requires the user to have logged into the GUI at least once since the last reboot (the GUI domain isn't initialized until login). SSH-on-fresh-boot installs fail with `Bootstrap failed: 5: Input/output error`. Documented in the README; no fallback to legacy syntax (workaround is "log into GUI, retry").
- `install/_common.detect_invocation()` is shared between `linux_systemd.py` and `macos_launchd.py`. Both lazy-import it inside their `install_*()` functions. When monkeypatching for tests, patch at the **import site** (`dbxignore.install.linux_systemd.detect_invocation` or `dbxignore.install.macos_launchd.detect_invocation`), not at the source (`_common.detect_invocation`) — `from ... import` creates a local binding in the consuming module, and that binding is what needs intercepting.
- `install/linux_systemd.py:_FORWARDED_ENV_VARS` propagates env vars from the install caller's shell into the unit's `[Service] Environment=` block (currently `DBXIGNORE_ROOT`, `XDG_STATE_HOME`). **Path-resolution vars must be forwarded** — without forwarding, the systemd-launched daemon's `state.user_state_dir()` / `roots.discover()` resolution disagrees with the user's shell tools (`dbxignore status`, `clear`, etc.) running in the same shell, producing silent split-state bugs. Tuning vars (`DBXIGNORE_LOG_LEVEL`, `DBXIGNORE_DEBOUNCE_*_MS`) stay un-forwarded — users override those via `dbxignore.service.d/` drop-ins.
- **systemd performs specifier expansion (`%h`, `%T`, `%u`, etc.) inside both `ExecStart=` AND `Environment=` values** — not just one. `_quote_exec_start_path` doubles `%` → `%%` for paths; `build_unit_content`'s env-line construction does the same for forwarded values. Any new var added to `_FORWARDED_ENV_VARS` whose value may contain a literal `%` (e.g. `XDG_STATE_HOME=/home/me/100%state`) needs the doubling applied before the C-style quoted-string escape — the two transformations are orthogonal.
- **macOS Dropbox sync modes:** `_backends/macos_xattr.py` writes one of two xattrs — `com.dropbox.ignored` (legacy Dropbox folder at `~/Dropbox`, synced by Dropbox's own daemon) or `com.apple.fileprovider.ignore#P` (File Provider mode at `~/Library/CloudStorage/Dropbox/`, synced by Apple's File Provider extension; default for installs since 2023). Detection logic (`_detect()` truth table, path-primary + pluginkit-disambiguating), dual-attribute fallback behavior, the `detection_summary()` cross-platform facade, `pluginkit -m -A -i <bundle-id>` output format, and the `_fake_pluginkit` / `_stage_dropbox_info` subprocess-mocking pattern are documented in the module docstring at `src/dbxignore/_backends/macos_xattr.py`.
- Subprocess-using code tests via `monkeypatch.setattr("subprocess.run", fake_fn)` rather than mocking the calling function. Template: `_fake_pluginkit` helper in `tests/test_macos_xattr_unit.py` — accepts `stdout=` for the success path or `side_effect=` for `FileNotFoundError`/`TimeoutExpired`. Future install-backend tests (schtasks, systemctl, launchctl) should use the same shape.
- Packaged non-.py data files (currently `src/dbxignore/templates/default.dropboxignore`, used by `cli.init`) ship automatically because hatchling's `packages = ["src/dbxignore"]` includes everything under the package directory, including subdirs and non-.py files. Access at runtime via `importlib.resources.files("dbxignore.<subpkg>").joinpath("<file>").read_text(encoding="utf-8")` — requires an `__init__.py` in the subpkg directory. Verify packaging on a fresh add by `uv build && python -c "import zipfile; z = zipfile.ZipFile(<wheel>); print([n for n in z.namelist() if '<subpkg>' in n])"` before relying on it.
- Tests asserting `dbxignore status` or `cli.clear` daemon-alive behavior should `monkeypatch.setattr(state, "is_daemon_alive", lambda pid: <bool>)` rather than relying on `daemon_pid=os.getpid()` matching the daemon-name guard. The pytest process name varies by host: Linux + `uv run pytest` shows up as `pytest`, Windows + `uv run python -m pytest` as `python.exe`, and `is_daemon_alive` only matches `"python"` or `"dbxignored"` substrings — Linux runs land in `state=not_running` if the test relies on `os.getpid()`.
- PowerShell `Set-Content -Encoding utf8` writes UTF-8-with-BOM on Windows PowerShell 5.1 but BOM-less on PS 7+. Any new PS script under `scripts/` should `#requires -Version 7.0` at the top — gates against the divergence with a clear error message rather than silently producing different file bytes (would break the byte-for-byte 4h `generate` assertion in `manual-test-windows.ps1` if the runner happened to launch under PS 5.1).
- Cross-platform `# type: ignore` codes differ by what's installed: `psutil` and `xattr` produce `[import-not-found]` on Windows (not installed) but `[import-untyped]` on Linux/macOS (installed, no stubs). Same shape for POSIX-only `os.getxattr`/`setxattr`/`removexattr`/`getuid` (`[attr-defined]` on Windows; the attr exists on Linux/macOS). Use the combined form `# type: ignore[import-not-found, import-untyped, unused-ignore]` (or `[attr-defined, unused-ignore]`) so the ignore covers whichever code fires AND silences `warn_unused_ignores` on the platforms where it doesn't.
- Two independent `dev` groups live in `pyproject.toml`: `[project.optional-dependencies].dev` (PEP 621 extras, ships in the wheel for `pip install dbxignore[dev]`) and `[dependency-groups].dev` (PEP 735, local-only). `uv sync --all-extras` hydrates both; plain `uv sync` only hydrates the second. CI uses `--all-extras`. Keep critical test-runtime deps (`pytest-timeout`, the conditional `xattr`) mirrored in both groups so `uv sync` plain doesn't ship a broken environment.
- Windows + Dropbox-synced working tree: `uv sync` / `uv pip install` editable-install steps hit `os error 32` ("file is being used by another process") on `.venv/Lib/site-packages/dbxignore-*.dist-info` while Dropbox is mid-sync. Workaround: `UV_LINK_MODE=copy` (env var or `--link-mode=copy`). The pre-commit hook's `uv run` invocation inherits the same problem; if the hook fails on the editable-install step (not on the actual lint/format/mypy check, which all show "Passed" in the hook output) it's the file lock, and `--no-verify` is acceptable for that specific commit when the manual `uv run` checks all pass cleanly. Same family as the existing `uv tool install` Cloud Files gotcha but for the local `.venv`.
- `--no-verify` skips ALL pre-commit hooks at once, including commit-check at the `commit-msg` stage — not just the lint/format/mypy hook that's failing on the file-lock above. When using `--no-verify`, run commit-check manually against each new commit subject before pushing — otherwise an over-72-char subject (or any other `cchk.toml` violation) lands locally and CI's commit-check action rejects the PR. CI runs commit-check across the full `origin/main..HEAD` range, so the workaround needs to validate every commit, not just `HEAD`. **Use the temp-file form, NOT `/dev/stdin`** — backlog item #83 documents that `commit-check -m /dev/stdin` is unreliable on Windows Git Bash in both loop and single-shot forms (reports on a stale subject from earlier in the session). Reliable shape: `for sha in $(git log origin/main..HEAD --format='%h'); do git log -1 --format='%s' $sha > /tmp/subj.txt && commit-check -m /tmp/subj.txt; done`.
- commit-check's `subject_max_length` counts BYTES, not visual characters. An em-dash (`—`) is 3 UTF-8 bytes, so each one burns 2 extra bytes against the 72-byte cap vs an ASCII `-`. A subject that visually fits the cap can still fail when it contains multi-byte chars; shorten or use ASCII `-` when budget-tight.
- **`isinstance(True, int) is True`** because Python's `bool` is an `int` subclass. JSON-decode numeric validation like `isinstance(value, (int, float))` therefore accepts hand-edited `"field": true`. Explicit bool exclusion is required: `if isinstance(value, bool) or not isinstance(value, (int, float)): raise ValueError(...)`. Used in `state.py::_validate_create_time`.
- **`msvcrt.locking` on Windows locks bytes from the file's current cursor position**, not a fixed offset (Python docs: "The lock starts at the file's current position"). Asymmetric with POSIX `fcntl.flock`, which is per-open-file (cursor-independent). For singleton-lock patterns: two concurrent processes opening the same lock file with `"ab+"` mode end at different cursor positions after any placeholder write and lock different byte ranges — both succeed, the singleton is defeated. Always `fh.seek(0)` immediately before `msvcrt.locking` so all contenders compete for byte 0; the seek is a no-op on POSIX but keeps the cross-platform contract uniform. Implemented in `daemon._acquire_singleton_lock`.
- `_logging.timed_debug(logger, msg_fmt, *args)` is the canonical context manager for DEBUG-level timing in hot paths (`reconcile._reconcile_path`, `rules.RuleCache.reload_file`). It gates the `time.perf_counter()` calls on `logger.isEnabledFor(DEBUG)` so the no-op-when-not-debug claim actually holds. Don't write raw `t0 = time.perf_counter(); ...; logger.debug(... time.perf_counter() - t0)` in the sweep loop — the original instrumentation paid the perf_counter cost on every call regardless of log level (measured 49.62s on a 27k-dir tree).
- `pytest-timeout` global is 10s (`pyproject.toml [tool.pytest.ini_options] timeout = 10`). Tests with poll loops longer than that — or with shutdown sequences that legitimately need more headroom — must override per-test via `@pytest.mark.timeout(N)`. Without the override, the test is killed at 10s before its own timeout fires.
- Daemon-thread tests using `BlockingMarkers` (gate-closed `is_ignored`) can deadlock on macOS. FSEvents emits synthetic events for files created just before `observer.schedule` (e.g. a test's `.dropboxignore`); those queue in the debouncer, fire `_dispatch` → `reconcile_subtree` → `markers.is_ignored` blocked on the gate. Then `debouncer.stop()` (no join timeout — deliberate design for clean in-flight emit completion) hangs forever, never reaching `worker.join` or the "did not exit" WARNING. Test fix: stub `daemon.Observer` to a no-op for tests that gate the worker before opening the gate. See `tests/test_daemon_singleton.py::test_singleton_lock_not_released_while_worker_alive` for the pattern. Linux + Windows don't synthesize events for existing files, so this is macOS-specific.
- `Path.resolve()` case behavior diverges by platform: Windows returns on-disk casing (so `WindowsPath` equality on resolved paths effectively normalizes case); POSIX preserves the queried casing — `Path('.dropboxignore').resolve() == Path('.DropboxIgnore').resolve()` is True on Windows but **False** on macOS APFS-default (case-insensitive FS), because `PosixPath` equality is case-sensitive regardless of FS case-sensitivity. Detect case-insensitive FS in tests by probing instead: write one casing, then check `alternate.exists()`.
- `Path.rglob('.dropboxignore')` and `os.walk(...)` filenames list differ on case-insensitive filesystems: `rglob` uses the FS's case-insensitive lookup at glob-query level (NTFS, default macOS APFS/HFS+ all match `.DropboxIgnore` for query `.dropboxignore`); `os.walk` returns on-disk casing exactly, so an exact-string `IGNORE_FILENAME in filenames` check misses on those FSes. Switching from `rglob` to `os.walk` requires explicit case-insensitive matching (`f.lower() == IGNORE_FILENAME`) PLUS canonical-key normalization (`<dir>/IGNORE_FILENAME` instead of the discovered path) PLUS prefer-exact-match selection when both casings exist.
- `commit-check -m <file>` reads the **entire file content** as the subject under check. In a per-commit verification loop, write ONLY the bare subject to the temp file: `git log -1 --format='%s' $sha > /tmp/subj.txt && commit-check -m /tmp/subj.txt`. Adding annotation prefixes (`echo "$sha [$BYTES] $SUBJ" > /tmp/subj.txt`) makes commit-check parse the SHA prefix as a non-Conventional-Commits type and reject every commit — easy to misdiagnose as "my subjects are bad" when the loop's input format is what's wrong.
- `tests/test_daemon_inotify_enospc.py`'s 3 daemon-startup tests fail locally on a dev machine that has a real `dbxignored` daemon running — the daemon's per-user `daemon.lock` holds the singleton, so `daemon.run` exits cleanly via the singleton-not-acquired path instead of raising the expected `OSError`.
