Maintenance release. Release-workflow hardening and project-documentation scaffolding.

**No user-facing behavior changes.** Existing `.dropboxignore` rules, CLI commands (`dropboxignore apply` / `status` / `list` / `explain` / `install` / `uninstall`), and daemon behavior all work identically to v0.2.0. Upgrade is a no-op for anyone on v0.2.0 today.

## What's new (for maintainers)

### Dry-runnable release workflow

`.github/workflows/release.yml` now accepts `workflow_dispatch` — the release build can be run on-demand via `gh workflow run release.yml` (or the GitHub Actions UI) without cutting a tag. The Publish step is gated on the ref being a tag, so dispatch runs validate the full build pipeline (wheel + sdist + Windows binaries via PyInstaller) without creating a Release object. Prevents the "first exercise is the actual release" failure mode that bit v0.2.0.

### Releases author under the repo owner's identity

When the repo's `GH_RELEASE_TOKEN` secret is set (fine-grained PAT with `Contents: Read and write`), releases attribute to the owner's GitHub identity instead of `github-actions[bot]`. Missing or expired secret falls through to the default token via a `||` fallback — no workflow breakage if the PAT isn't configured.

### Project-documentation scaffolding

- `CHANGELOG.md` at repo root — [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) format, retrospective v0.1.0 and v0.2.0 entries alongside this one.
- `docs/release-notes/v<X.Y.Z>.md` convention — each release's hand-crafted body is versioned alongside its tag, applied via `gh release edit <tag> --notes-file ...` after the workflow publishes.
- CLAUDE.md additions: pre-flight commit-check snippet matching CI's per-commit validation, `hatch-vcs` auto-versioning behavior, SemVer stance pre-1.0, and the Keep-a-Changelog + release-notes conventions.
