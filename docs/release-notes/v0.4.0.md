# dbxignore v0.4.0 — macOS support

dbxignore now runs on macOS — third platform after Windows and Linux.

## What's new for macOS users

dbxignore on macOS auto-detects which Dropbox sync mode is active and writes the matching extended attribute. Both modes are supported:

- **Legacy mode** — Dropbox folder at `~/Dropbox`, ignored files marked via `com.dropbox.ignored`. Synced by Dropbox's own daemon.
- **File Provider mode** — Dropbox folder at `~/Library/CloudStorage/Dropbox/` (or `/Volumes/...` for the eligibility-gated external-drive variant), ignored files marked via `com.apple.fileprovider.ignore#P`. Synced by Apple's File Provider extension; the default for Dropbox installs since 2023.

Detection happens at module-load time and is cached for the process lifetime. The daemon registers as a launchd User Agent so it auto-starts at GUI login.

```bash
pip install dbxignore
dbxignore install
```

Or grab pre-built arm64 binaries from this Release page and `xattr -d com.apple.quarantine` them to bypass Gatekeeper. Intel Mac users go via PyPI — the wheel is universal Python. Full instructions in the [macOS install section of the README](https://github.com/kiloscheffer/dbxignore#install-macos).

A few things specific to macOS:

- **Files split between two dirs.** State (`state.json`) lives in `~/Library/Application Support/dbxignore/`; logs (`daemon.log`, `launchd.log`) live in `~/Library/Logs/dbxignore/`. Matches Apple's app-data conventions.
- **Symlinks get marked on the link itself, not the target.** The macOS kernel allows xattrs on symlinks via the NOFOLLOW path, so the marker lands silently. (Linux symlinks cannot carry `user.*` xattrs at all — kernel restriction; the Linux backend emits a WARNING and skips them.)
- **First install needs a GUI login since boot.** `dbxignore install` bootstraps into the user's GUI domain; SSH-on-fresh-boot users will see `Bootstrap failed: 5: Input/output error`. Log into the GUI once, retry.

## What's new for Windows / Linux users

**No breaking changes.** The only user-visible delta is that the project description and README now mention macOS. State directories, command behavior, marker mechanism, daemon installation — all unchanged.

## Caveats

- **arm64 only for pre-built binaries.** Intel Mac users install via PyPI. If x86_64 demand surfaces, a point release will add an `macos-13` build leg.
- **Beta-validated, not field-validated at scale.** v0.4.0 was validated on real hardware by a beta tester (macOS Tahoe 26.4 / Dropbox 250.4, File Provider mode) before this release, but lacks the multi-user shake-out that Windows and Linux have accumulated. Please file issues for anything unexpected.

## The alpha cycle

v0.4.0 ships after five alphas (`a1` through `a5`), each addressing a layer of issue exposed by beta-testing or by reasoning about the prior alpha's logic:

- `a3` fixed cffi bundling (`ModuleNotFoundError: No module named '_cffi_backend'` on first launch of the macOS binary).
- `a4` fixed the launchd plist invocation target (the daemon was never starting because the plist ran the long-form CLI without a subcommand) and added the first File Provider auto-detection.
- `a5` refined detection to be path-primary (info.json's `path` field) with pluginkit-disambiguation, fixing a system-level-vs-user-level signal conflation that would have misdetected sync mode for users with Dropbox.app installed but who had declined the File Provider migration.

`a5` passed the beta tester's pass (sandboxing entitlements, stub-file behavior, `#P`-vs-`#N` semantics all answered as "works as designed"), promoting to v0.4.0 final.
