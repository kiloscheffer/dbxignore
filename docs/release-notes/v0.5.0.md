# dbxignore v0.5.0 ‚Äî reliability sweep + new path-taking verbs

The headline shape of v0.5.0: **`dbxignore ignore <path>` and `dbxignore unignore <path>`** for per-path rule management, plus a reliability sweep across destructive verbs, the rule cache, and the path validator. Several Breaking callouts ‚Äî see the upgrade notes below if you script against the CLI.

## New: `ignore` and `unignore` verbs

Two new verbs for the common case of "ignore this one path" without hand-editing `.dropboxignore`:

```bash
dbxignore ignore ~/Dropbox/Project/build/
dbxignore unignore ~/Dropbox/Project/build/
```

`ignore` selects the closest `.dropboxignore` ancestor (or creates one at the Dropbox root), appends a gitignore-anchored rule matching the target literally, then sets the marker. `unignore` does the inverse. Wildcard collisions on `unignore` fail loud with a friendly error naming the blocking rule.

Both verbs play nicely with the existing `apply` / `clear` / `list` / `explain` / `check-ignore` surface, all of which now also accept path arguments through the same validator.

## Breaking changes

**Path-taking verbs operate on the symlink object, not the target.** `apply ~/Dropbox/link-to-external/` previously called `path.resolve()` and operated on the external target tree ‚Äî the marker would land outside any Dropbox root, and the daemon's own walk (with `followlinks=False`) couldn't reach it to maintain. The new behavior preserves the link object: the marker attaches to the link itself (silently on macOS/Windows, with a WARNING on Linux where the kernel refuses `user.*` xattrs on symlinks). Filesystem-state verbs additionally refuse paths with a symlinked **ancestor** (the marker would be stranded), and verbs reject `link/../file` shapes where lexical `..`-collapse diverges from filesystem-true resolution.

**`uninstall --purge` reports marker errors and exits 2 on partial failure.** Previously the marker loop caught every `OSError` with `pass` and unconditionally reported `Cleared N`. Now per-path failures are listed on stderr (`Could not fully clear markers (N errors):` plus bounded entries naming each path and the operation that failed), and exit code is 2 when any marker error occurred. Scripts consuming the happy-path output stay compatible; scripts that conditionally branched on `dbxignore uninstall --purge` exiting 0 should now handle the 2-on-partial case explicitly.

**`clear` fails closed when `state.json` is present but unreadable.** Previously `clear`'s daemon-alive guard silently skipped on `state.json` errors, then proceeded to wipe markers that a live daemon would immediately re-apply. Now `clear` exits 2 with a "daemon liveness is unknown" message in that case. Pass `--force` to override (the same flag that overrides the daemon-alive guard).

**`status --summary` emits a fourth state token `state=starting`** while the daemon is alive but the initial sweep hasn't yet completed. Consumers branching exhaustively on `state == "running"` need to handle the new value. During the starting window, the summary line contains only `state` and `pid` ‚Äî the other fields would all be 0 and falsely imply a completed sweep.

## Reliability fixes

- **Rule cache catches same-size content swaps with preserved mtimes.** The cache-invalidation gate switched from `(mtime_ns, size)` stat to a blake2b-128 hash of the file's bytes. Tools like `touch -r` and some editors restore the original mtime after a save; the previous heuristic missed those edits.
- **`.dropboxignore` files with invalid UTF-8 no longer crash the daemon's sweep.** A decode error is now treated like a parse error: log a warning, drop the cached entry, treat the file as empty until the next valid edit.
- **CLI rule-file writes use unique temp names and preserve file permissions.** Two concurrent `ignore` invocations no longer collide on `<rule_file>.tmp`; group-readable `.dropboxignore` files no longer get silently relocked to user-only after the first CLI write.
- **`dbxignore status` and `clear` no longer crash on a locked/unreadable `state.json`** (formerly raised `OSError` straight through to the user).
- **`uninstall --purge` skips vanished paths silently.** A path listed by `os.walk` that disappears before our `is_ignored` / `clear_ignored` call (Dropbox sync, IDE temp moves) no longer registers as a partial-failure error.
- **`clear`, `list`, and `uninstall --purge` now also clear the marker on the target/root itself.** Previously the implicit `os.walk` entry skipped the top-level path; a marker on a Dropbox root or on a directory passed as an argument was silently missed.

## Upgrade notes

If you script against the CLI:

- Audit any pipeline that passes paths containing `..` after a symlinked component ‚Äî those now exit 2 instead of silently following the lexical interpretation. Resolve the path beforehand (`readlink -f` / `Get-Item -Path X | Resolve-Path`) or rewrite to avoid the `..`.
- Audit any pipeline that consumes `dbxignore uninstall --purge` ‚Äî exit code 2 is now possible on partial failure (existing happy-path exit 0 unchanged).
- Audit any pipeline that branches on `dbxignore status --summary` output ‚Äî handle the new `state=starting` token.
- Audit any pipeline that runs `dbxignore clear` against a tree where `state.json` might be locked or inaccessible ‚Äî exit code 2 is now possible (formerly an uncaught `OSError`); pass `--force` if your script knows no daemon is alive.

## Internal

- CI now uses the documented canonical `uv run python -m pytest` invocation (was `uv run pytest`).
- Manual-test scripts (`scripts/manual-test-{ubuntu-vps,macos,windows}.{sh,ps1}`) extended with two new Phase 4.5 cases covering the new CLI surface; AGENTS.md's manual-test-extension rule strengthened from descriptive to a hard requirement at PR-time.
