# dbxignore v0.5.1 — Windows LF-pin fix for `generate` / `init`

Patch release for two `cli.write_text` callsites in v0.5.0 that inherited Python's `newline=None` default and translated `\n` → `\r\n` on Windows. No CLI surface changes; no breaking changes.

## Fixed

- **`dbxignore generate` byte-for-byte invariant on Windows.** When the source `.gitignore` was LF-canonical, the generated `.dropboxignore` came out with CRLF on Windows installs. The byte-for-byte contract documented in v0.5.0's release notes was broken for that case. `cli.generate` now passes `newline=""` to its write so the output bytes match the input bytes regardless of platform.
- **`dbxignore init` LF-canonical output on Windows.** v0.5.0 documented the `init`-written `.dropboxignore` as LF-canonical on every platform (so a file committed-via-`init` from Windows and read by a teammate on Linux is byte-identical). The contract regressed silently on Windows after PR #207; restored.

Both fixes are one-line `newline=""` additions on the corresponding `write_text` calls in `src/dbxignore/cli.py`. No state-file, daemon, or marker-write behavior changed.

## Other changes

- **README install sections** now end with `dbxignore status` as the cross-platform verification step on Windows, Linux, and macOS. Previously only Linux had a verification line (`systemctl --user status dbxignore.service`), so a user scrolling for "did this work" from the Windows section could land on the wrong command. Linux retains `systemctl --user status` and `journalctl --user -u dbxignore.service` as a follow-up paragraph for systemd-level unit state and log output.

## Internal

Several manual-test-script fixes surfaced during the v0.5.0 release-validation pass:

- Phase 4 `apply` calls in `scripts/manual-test-{ubuntu-vps,macos}.sh` now pass `--yes` (previously hung on the confirmation prompt).
- Phase 5f's human-status `grep -q` check now captures output into a variable first, then greps the variable — `set -o pipefail` plus a multi-line click producer plus `grep -q` exiting on first match was producing false failures (the producer's next `click.echo` hit a closed pipe → `BrokenPipeError` → Python exits 1 → pipefail propagates).
- Windows Phase 5 case 5e narrowed to a single file (the prior `Reset-TestDir` wiped the tree before Phase 6's marker-retention assertion).
- Case 4m target renamed from `build/` to `case4m_target/` so the user's ancestor `.dropboxignore` rule (`build/` from the `init` template) doesn't mask the test's local `!build/keep/` negation.
- Windows `Reset-TestDir` retries on transient file-lock contention.

Three new gotcha entries in `docs/internals/active-gotchas.md` (SIGPIPE+pipefail trap with `grep -q`; `pre-commit install` needs explicit hook-type flags; uv tool cache can serve a stale wheel after uninstall).
