# dbxignore v0.4 — macOS support

**Status:** Design approved 2026-04-26.
**Scope:** Add macOS as a third supported platform alongside Windows and Linux.
**Validation strategy:** Manual beta-test by a recruited tester on a real Dropbox tree, then promote `v0.4.0a1` → `v0.4.0`. No PyPI publication until promotion.

## Context

dbxignore currently supports two platforms: Windows (NTFS Alternate Data Streams) and Linux (`user.com.dropbox.ignored` extended attributes). The architecture was built from the start with platform dispatch in mind — `_backends/` directory, `markers.py` import-time dispatch, `install/` package with sibling-module installers, `state.user_state_dir()` branching. Adding macOS is the architecture's first real test of "third platform fits in without rework."

The maintainer has no Mac available locally, which constrains validation:

- All work must be designed against documentation and remote CI.
- `macos-latest` GitHub Actions runs are the only automated check until release.
- Final release-quality validation must come from a recruited beta tester running the pre-release on real hardware.

## Goals

1. Mac users can `pip install dbxignore`, run `dbxignore install`, and have ignore markers maintained on their Dropbox tree the same way Windows and Linux users do.
2. The marker mechanism uses Dropbox's documented macOS protocol — the `com.dropbox.ignored` extended attribute. Same value semantics as Linux: any non-empty byte value marks the path as ignored.
3. The daemon installs as a launchd User Agent (`~/Library/LaunchAgents/com.kiloscheffer.dbxignore.plist`) and starts at GUI login.
4. macOS support reaches the same test discipline as Windows and Linux: a `macos_only` pytest marker, integration tests on the `macos-latest` runner, the existing test patterns (module-level guards, `tmp_path`-rooted fixtures) extended to the third platform.
5. Pre-built arm64 Mach-O binaries (`dbxignore`, `dbxignored`) ship on the GitHub Release for users who don't want to install Python.
6. v0.4.0 ships to PyPI **only** after a beta tester has validated the pre-release on real hardware against a real Dropbox tree.

## Non-goals (v0.4)

- **x86_64 (Intel) Mach-O binaries.** Intel users install via `pip install dbxignore` (the wheel is universal Python). If demand surfaces post-release, file as a backlog item and add an `macos-13` build leg in a point release.
- **Universal2 binaries.** Possible via `target_arch="universal2"` but adds complexity; defer until arm64 + x86_64 split is proven inadequate.
- **Codesigning / notarization.** Requires an Apple Developer account ($99/yr). Pre-built binaries are unsigned; users bypass Gatekeeper via `xattr -d com.apple.quarantine` (documented in README). If complaints surface, revisit.
- **Migration tooling for hypothetical pre-existing macOS state.** No prior macOS release exists, so there's no on-disk state to migrate.
- **Anything outside macOS introduction.** No Windows/Linux refactors, no unrelated cleanup, no backlog items beyond what naturally surfaces in the touched files.

### Invariants preserved

- `markers.{is,set,clear}_ignored` keep absolute-path-required `ValueError` contract on macOS.
- The reconcile `OSError(ENOTSUP|EOPNOTSUPP)` arm continues to handle filesystems that don't support xattrs — same errno values as Linux.
- The `.dropboxignore` rule file itself is never marked ignored — existing guards in `match()` / `explain()` / `reconcile._reconcile_path` are platform-agnostic.
- Symlink discipline: macOS xattr calls use `XATTR_NOFOLLOW`, mirroring Linux's `follow_symlinks=False`. **One intentional behavioral divergence** — see "Symlink semantics" below.

### What does not change

- `rules.py`, `rules_conflicts.py`, `debounce.py` — pure logic, platform-blind.
- The watchdog event flow — `watchdog` already supports macOS via FSEvents; no opt-in needed beyond the PyInstaller hidden import.
- `_backends/linux_xattr.py`, `_backends/windows_ads.py` — untouched.

## macOS xattr backend (`_backends/macos_xattr.py`)

Mirrors `linux_xattr.py` shape-for-shape. Key differences are the API surface (`xattr` PyPI package vs stdlib `os.*xattr`), the attribute name (no `user.` prefix), and one BSD-vs-Linux errno.

### Public API

```python
ATTR_NAME = "com.dropbox.ignored"   # macOS xattrs have no namespace prefix
_MARKER_VALUE = b"1"

def is_ignored(path: Path) -> bool: ...
def set_ignored(path: Path) -> None: ...
def clear_ignored(path: Path) -> None: ...
```

### Implementation

```python
import errno, logging
from pathlib import Path
import xattr

logger = logging.getLogger(__name__)

ATTR_NAME = "com.dropbox.ignored"
_MARKER_VALUE = b"1"

# macOS errno.ENOATTR (93). Python on Linux doesn't define ENOATTR
# (Linux uses ENODATA). Defensive getattr guards an accidental import
# on Linux during cross-platform unit tests.
_NO_ATTR_ERRNO = getattr(errno, "ENOATTR", 93)

def _require_absolute(path: Path) -> None:
    if not path.is_absolute():
        raise ValueError(f"markers requires an absolute path; got {path!r}")

def is_ignored(path: Path) -> bool:
    _require_absolute(path)
    try:
        value = xattr.getxattr(str(path), ATTR_NAME, options=xattr.XATTR_NOFOLLOW)
    except OSError as exc:
        if exc.errno == _NO_ATTR_ERRNO:
            return False
        if exc.errno == errno.ENOENT:
            raise FileNotFoundError(str(path)) from exc
        raise
    return bool(value)

def set_ignored(path: Path) -> None:
    _require_absolute(path)
    xattr.setxattr(str(path), ATTR_NAME, _MARKER_VALUE, options=xattr.XATTR_NOFOLLOW)

def clear_ignored(path: Path) -> None:
    _require_absolute(path)
    try:
        xattr.removexattr(str(path), ATTR_NAME, options=xattr.XATTR_NOFOLLOW)
    except OSError as exc:
        if exc.errno == _NO_ATTR_ERRNO:
            logger.debug("clear_ignored: xattr absent on %s", path)
            return
        if exc.errno == errno.ENOENT:
            logger.debug("clear_ignored: path gone: %s", path)
            return
        raise
```

### Errno table

| Errno | Linux meaning | macOS meaning | Handling |
| --- | --- | --- | --- |
| `ENODATA` (61) | xattr absent | not used | not on this code path |
| `ENOATTR` (93 on macOS) | not on Linux | xattr absent | `is_ignored` → `False`; `clear_ignored` → no-op |
| `ENOTSUP` / `EOPNOTSUPP` | fs doesn't support xattrs | same | propagates → `reconcile._reconcile_path` `OSError` arm catches |
| `ENOENT` | path gone | same | `is_ignored` → `FileNotFoundError`; `clear_ignored` → debug-log + no-op |
| `EPERM` on symlink | kernel refuses `user.*` on symlinks | doesn't fire — `XATTR_NOFOLLOW` allows | macOS marks symlinks; Linux's `PermissionError` arm in reconcile is dormant on macOS |

### Symlink semantics — one intentional divergence

CLAUDE.md documents the design intent as "a symlink marked ignored means the link itself is marked, not its target." Linux can't deliver this — the kernel refuses `user.*` xattrs on symlinks with `EPERM`, and `_reconcile_path`'s `PermissionError` arm logs WARNING. macOS *can* deliver it (no kernel restriction with `XATTR_NOFOLLOW`), so the symlink itself gets marked silently. This is **better behavior**, not a bug — but it's user-visible. The divergence is documented in three places: CLAUDE.md (new gotcha), the CHANGELOG `[0.4.0]` Caveats section, and the README's macOS install section.

### `roots._info_json_path()` — darwin folds into Linux arm

```python
if sys.platform.startswith("linux") or sys.platform == "darwin":
    home = os.environ.get("HOME")
    if not home:
        logger.warning("HOME not set; cannot locate Dropbox info.json")
        return None
    return Path(home) / ".dropbox" / "info.json"
```

Folded because the *behavior* is identical — Dropbox client puts `info.json` at `~/.dropbox/info.json` on both platforms with identical JSON shape.

### `state.py` — split state dir vs log dir

The Q4=B path-conventions choice (`~/Library/Application Support/` for state, `~/Library/Logs/` for logs) means `state.json` and `daemon.log` no longer live in the same directory on macOS. Add a sibling function `user_log_dir()`:

```python
def user_state_dir() -> Path:
    """Per-user directory for state.json (and daemon.log on Linux/Windows)."""
    if sys.platform == "win32":
        ...
    if sys.platform == "darwin":
        return Path.home() / "Library" / "Application Support" / "dbxignore"
    # Linux: $XDG_STATE_HOME or ~/.local/state, then /dbxignore
    ...

def user_log_dir() -> Path:
    """Per-user directory for daemon.log (split from state on macOS only)."""
    if sys.platform == "darwin":
        return Path.home() / "Library" / "Logs" / "dbxignore"
    return user_state_dir()                           # Windows + Linux: combined
```

Branched here (rather than folded) because the *behavior* diverges. `daemon._log_dir()` becomes a one-line passthrough to `state.user_log_dir()`.

### `cli._purge_local_state()` — clean both dirs on macOS

`uninstall --purge` previously cleared `user_state_dir()` only. On macOS it must also clear `user_log_dir()`. Restructured to a per-dir helper:

```python
def _purge_local_state() -> None:
    _purge_dir(state.user_state_dir(), patterns=["state.json", "state.json.tmp",
                                                  "daemon.log", "daemon.log.*"])
    if sys.platform == "darwin":
        _purge_dir(state.user_log_dir(), patterns=["daemon.log", "daemon.log.*",
                                                    "launchd.log"])
```

`launchd.log` is the launchd-captured stdout/stderr file (see "launchd installer" below). Two regression tests cover the macOS purge path: one `macos_only`, one cross-platform with mocked `state.user_log_dir`.

## launchd installer (`install/macos_launchd.py`)

macOS peer of `linux_systemd.py`. Same public surface (`install_agent` / `uninstall_agent`), same env-var-forwarding (`_FORWARDED_ENV_VARS = ("DBXIGNORE_ROOT",)`), same `RuntimeError`-on-failure contract.

### Constants

```python
LABEL = "com.kiloscheffer.dbxignore"
PLIST_FILENAME = f"{LABEL}.plist"
_FORWARDED_ENV_VARS = ("DBXIGNORE_ROOT",)
```

### Plist generation — `plistlib`, not string formatting

```python
def build_plist_content(
    label: str,
    program_arguments: list[str],
    log_dir: Path,
    environment: dict[str, str] | None = None,
) -> bytes:
    plist: dict = {
        "Label": label,
        "ProgramArguments": program_arguments,
        "RunAtLoad": True,
        "KeepAlive": {"SuccessfulExit": False, "Crashed": True},
        "StandardOutPath": str(log_dir / "launchd.log"),
        "StandardErrorPath": str(log_dir / "launchd.log"),
    }
    if environment:
        plist["EnvironmentVariables"] = environment
    return plistlib.dumps(plist)
```

`KeepAlive: {SuccessfulExit: False, Crashed: True}` matches the spirit of systemd's `Restart=on-failure` — restart on non-zero exit or signal kill, but respect a clean unload. launchd's built-in 10s throttle covers what systemd's `RestartSec=60s` does explicitly.

### Modern `launchctl bootstrap` / `bootout` — not legacy `load -w`

```python
def _domain() -> str:
    return f"gui/{os.getuid()}"

def _service_target(label: str = LABEL) -> str:
    return f"{_domain()}/{label}"

def install_agent() -> None:
    exe, args = detect_invocation()
    program_args = [str(exe), *(args.split() if args else [])]
    environment = {n: os.environ[n] for n in _FORWARDED_ENV_VARS if os.environ.get(n)}
    log_dir = state_module.user_log_dir()
    log_dir.mkdir(parents=True, exist_ok=True)

    content = build_plist_content(LABEL, program_args, log_dir, environment or None)
    path = _plist_path()
    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_bytes(content)

    # Bootout an existing instance first; bootstrap fails on duplicate label.
    subprocess.run(
        ["launchctl", "bootout", _service_target()],
        check=False, capture_output=True,
    )
    _run_launchctl(["launchctl", "bootstrap", _domain(), str(path)])

def uninstall_agent() -> None:
    subprocess.run(
        ["launchctl", "bootout", _service_target()],
        check=False, capture_output=True,
    )
    path = _plist_path()
    if path.exists():
        path.unlink()
```

### GUI-domain prerequisite

`gui/<uid>` requires the user has logged into the macOS GUI at least once since the last reboot. SSH-on-fresh-boot installs fail with `Bootstrap failed: 5: Input/output error`. Documented in README. No fallback to legacy `load -w` — the workaround (log into GUI, retry) is trivial, and maintaining two code paths isn't worth it.

### `detect_invocation()` extraction

The Linux logic for finding `dbxignored` (PyInstaller frozen → PATH shim → `python3 -m dbxignore daemon` fallback) applies on macOS too. Extract once to `install/_common.py`; both platform installers import it. Only refactor of existing platform code in v0.4. Small (~15 lines moved, 2 imports), reviewable, eliminates a copy-paste before it can drift.

### No live `launchctl` integration test on `macos-latest` CI

The runner has no GUI session, and `launchctl bootstrap gui/<uid>` requires one. Beta tester does live install/uninstall validation; CI exercises plist generation only via `tests/test_macos_launchd.py` (`plistlib.loads` round-trip).

### What CI catches vs. what it defers to the beta tester

**Catches automatically on `macos-latest`:**

- The `xattr` package's actual API matches what we wrote.
- `is_ignored` / `set_ignored` / `clear_ignored` round-trips on a real APFS volume.
- `XATTR_NOFOLLOW` is correctly named.
- `errno.ENOATTR` (93) is the actual return code for "attribute absent."
- Plist generation is byte-equivalent to `plistlib.dumps` (round-trip in unit test).
- macOS path conventions work without permissions surprises in `tmp_path`-rooted trees.

**Defers to beta tester:**

- `launchctl bootstrap gui/<uid>` actually loads the agent (CI runner has no GUI session).
- FSEvents fire under real-Dropbox-tree workloads (CI runner has no Dropbox client).
- `~/.dropbox/info.json` parsing on a real Dropbox install.
- Symlink-marking on a symlink that crosses an actual Dropbox folder boundary.
- Daemon survives sleep/wake cycles.
- Reboot auto-start works.

## PyInstaller spec + release workflow

### `pyinstaller/dbxignore-macos.spec` — separate file

Sibling to `dbxignore.spec` (Windows) rather than parameterized. Specs are configuration; branching `sys.platform` inside obscures the platform-specific assumptions a maintainer needs visible when debugging. The platform-specific bits (`hiddenimports`, `console=`, binary names, UPX behavior) all differ enough to warrant a sibling.

```python
# pyinstaller/dbxignore-macos.spec
from pathlib import Path

SRC = Path("src").resolve()
ENTRY = SRC / "dbxignore" / "__main__.py"


def _analysis():
    return Analysis(
        [str(ENTRY)],
        pathex=[str(SRC)],
        binaries=[],
        datas=[],
        # FSEvents is the macOS watchdog backend. hiddenimports forces
        # PyInstaller's analyzer to bundle it since the import is dynamic
        # via watchdog's platform-detection layer.
        hiddenimports=["watchdog.observers.fsevents"],
        hookspath=[],
        hooksconfig={},
        runtime_hooks=[],
        excludes=[],
        cipher=None,
        noarchive=False,
    )


for variant_name in ("dbxignore", "dbxignored"):
    a = _analysis()
    pyz = PYZ(a.pure, a.zipped_data, cipher=None)
    exe = EXE(
        pyz, a.scripts, a.binaries, a.zipfiles, a.datas, [],
        name=variant_name,
        debug=False,
        strip=False,
        upx=False,                       # UPX-compressed Mach-O confuses Gatekeeper
        runtime_tmpdir=None,
        target_arch=None,                # arch follows the runner (arm64)
        codesign_identity=None,          # unsigned
        entitlements_file=None,
    )
```

(Ergonomic note: written as a loop above for spec brevity; can also be unrolled into two explicit `EXE(...)` blocks if PyInstaller's spec parser gets unhappy with the loop form. The Windows spec is unrolled; macOS spec can mirror that if needed.)

## Beta-test workflow

After PR 6 lands on main:

1. Land tag-prep (or amend the docs PR per maintainer preference): bump CHANGELOG `[Unreleased]` → `[0.4.0a1] — <date>` (date is the day the alpha tag is pushed; left as a placeholder here because the calendar gap depends on PR cadence and beta-tester recruitment).
2. Push tag `v0.4.0a1`. `release.yml` fires; `publish-github` marks pre-release; `publish-pypi` waits at the `pypi` environment gate — **don't approve**.
3. Hand the beta tester the GitHub Release URL.
4. Tester runs the checklist (below). Anything diverging → fix PR(s), bump to `v0.4.0a2`, repeat.

### Beta-test checklist

1. **Install via Mach-O binary.**
   ```bash
   curl -L -o dbxignore  https://github.com/kiloscheffer/dbxignore/releases/download/v0.4.0a1/dbxignore
   curl -L -o dbxignored https://github.com/kiloscheffer/dbxignore/releases/download/v0.4.0a1/dbxignored
   chmod +x dbxignore dbxignored
   xattr -d com.apple.quarantine dbxignore dbxignored
   sudo mv dbxignore dbxignored /usr/local/bin/
   dbxignore --version    # expects: 0.4.0a1
   ```
2. **Install daemon agent.**
   ```bash
   dbxignore install
   launchctl print gui/$(id -u)/com.kiloscheffer.dbxignore   # state = running
   ls -la ~/Library/LaunchAgents/com.kiloscheffer.dbxignore.plist
   ```
3. **Apply a rule.**
   ```bash
   cd ~/Dropbox
   echo 'build/' > .dropboxignore
   mkdir -p build/keep
   dbxignore apply
   xattr -p com.dropbox.ignored build/        # 1
   xattr -p com.dropbox.ignored build/keep/   # 1
   ```
4. **Daemon picks up rule edits.**
   ```bash
   echo '!build/keep/' >> .dropboxignore
   sleep 5
   tail -20 ~/Library/Logs/dbxignore/daemon.log     # RULES event + reconcile
   xattr -p com.dropbox.ignored build/keep/         # still 1 (negation dropped per item-7 invariant)
   ```
5. **Daemon picks up new directories.**
   ```bash
   mkdir build/another
   sleep 5
   xattr -p com.dropbox.ignored build/another/      # 1
   ```
6. **Symlink marking.**
   ```bash
   ln -s /tmp/something build/symlink-target
   sleep 5
   xattr -p com.dropbox.ignored build/symlink-target  # 1 (macOS allows; Linux refuses)
   ```
7. **`dbxignore status`** — daemon-running, last sweep recent, no errors.
8. **Reboot, log into GUI, wait 30s, verify auto-start** via `launchctl print`.
9. **Uninstall (no purge).**
   ```bash
   dbxignore uninstall
   # plist gone, agent unloaded, markers preserved
   ```
10. **Reinstall + `uninstall --purge`** — markers cleared, both dirs gone.

### Acceptance criteria for promotion

- All 10 checklist steps complete with expected outputs.
- No crashes in `daemon.log`.
- No unexpected Gatekeeper dialogs after the documented `xattr -d com.apple.quarantine`.
- Daemon survives at least one sleep/wake cycle.
- Tester is willing to vouch.

### Promotion to `v0.4.0`

1. Land PR rolling CHANGELOG `[0.4.0a1]` → `[0.4.0]` with final date.
2. Push tag `v0.4.0`. `release.yml` fires; `publish-github` posts the real release; `publish-pypi` waits at the `pypi` env gate — **approve**.
3. `gh release edit v0.4.0 --notes-file docs/release-notes/v0.4.0.md` to swap in the hand-crafted body.
4. File `chore/v0.4-followups` PR with deferred BACKLOG items.

## Risks and open questions

1. **`xattr` package's exact API.** Spec assumes `xattr.getxattr(path, name, options=...)` with `xattr.XATTR_NOFOLLOW` constant. CI on `macos-latest` will catch any deviation immediately on PR 2.
2. **`launchctl bootstrap` on the GitHub runner.** No live integration test. Beta tester is the only validation path. If `bootstrap` fails on the tester's Mac for an unexpected reason (e.g. SIP-related interaction), v0.4 may need a fallback to `load -w` despite earlier reasoning.
3. **Beta-tester availability.** Recruitment is out of scope for the spec; the alpha tag can sit on GitHub indefinitely without bit-rotting (we can bump it forward to incorporate any unrelated `0.3.x` patches). No calendar pressure.
4. **macOS Sequoia / Tahoe-era launchd changes.** Apple has been progressively tightening LaunchAgent permissions and quarantine attributes in recent macOS versions. The spec targets currently-shipping behavior; if a future macOS version breaks what we ship, that's a point-release problem.
5. **FSEvents under Dropbox-client load.** The watchdog library uses FSEvents on macOS. Heavy Dropbox-client activity (large sync) generates lots of events; debouncer behavior under that load is unverified except by the beta tester.

## Post-ship backlog candidates

To be filed in `BACKLOG.md` via a separate `chore/v0.4-followups` PR after promotion, not as part of v0.4 itself:

- **x86_64 (Intel) Mach-O binary build leg.** Add `macos-13` runner to `release.yml`. Touches `pyinstaller/dbxignore-macos.spec`, `release.yml`. Urgency: low until demand surfaces.
